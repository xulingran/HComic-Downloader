# jm-interactive-challenge-recovery 规范

## 目的
定义 JM 收藏夹遇到持续 Cloudflare 人机验证时的浏览器交互恢复行为，包括挑战窗口、凭据同步、一次性自动重试及 Chromium 会话页面快照兜底。
## 需求
### 需求:用户前台请求可启动 JM 人机验证窗口

当用户主动加载、刷新或翻页 JM 收藏夹，且后台有界恢复仍返回反爬挑战时，系统必须打开单个模态浏览器窗口并直接加载受挑战的收藏夹 URL，让用户完成验证；禁止以 JM 首页替代受挑战路径。

#### 场景:主动加载触发验证
- **当** 用户主动加载 JM 收藏夹且收到结构化挑战信号
- **那么** 系统打开标题和叠层均表达“人机验证”语义的模态窗口
- **且** 窗口初始 URL 为本次受挑战的收藏夹 URL

#### 场景:后台请求不打扰用户
- **当** 缓存后台刷新、相邻页预加载或非交互工具调用收到挑战信号
- **那么** 系统禁止打开或聚焦验证窗口
- **且** 调用按可恢复挑战错误结束

### 需求:验证窗口必须安全限制目标和并发

验证窗口必须只接受 HTTPS JM 可信域收藏夹 URL，并与普通登录窗口共享单实例/单飞约束；非法 URL 必须在加载前拒绝。

#### 场景:拒绝非法挑战 URL
- **当** 挑战载荷包含非 HTTPS、用户信息、非默认端口、非 JM 白名单域名或非收藏夹路径的 URL
- **那么** 主进程拒绝打开窗口并返回安全错误
- **且** 禁止向该 URL 发出请求

#### 场景:并发挑战复用窗口
- **当** 一个 JM 验证窗口尚未结束时又有允许交互的请求收到挑战
- **那么** 系统复用同一恢复 Promise 并聚焦已有窗口
- **且** 禁止创建第二个窗口或覆盖叠层 IPC handler

#### 场景:登录与验证窗口互斥
- **当** 普通来源登录窗口与 JM 挑战窗口的打开请求发生重叠
- **那么** 系统必须串行化或复用当前来源窗口
- **且** 任意时刻最多存在一个使用登录叠层 IPC 通道的窗口

### 需求:验证成功必须同步浏览器认证上下文

用户完成验证后，系统必须从该 BrowserWindow 的 Session 提取目标域全部 Cookie 和实际 User-Agent，并通过主进程直接应用到 Python JM parser；Cookie 值禁止进入 React renderer 或日志。

#### 场景:同步 clearance 和登录 Cookie
- **当** 用户完成验证且目标域同时存在 JM 登录 Cookie 与 `cf_clearance`
- **那么** 系统将两类 Cookie 与窗口 User-Agent 一并回写 Python 认证配置
- **且** 后续 Python 请求复用现有系统代理 Session

#### 场景:浏览器缺少登录态
- **当** 验证窗口 Session 不含 JM 登录 Cookie并跳转到登录页
- **那么** 窗口保持打开并允许用户完成 JM 登录
- **且** 登录和挑战完成后一次性同步新的登录 Cookie、clearance Cookie 与 User-Agent

#### 场景:敏感信息不进入渲染端
- **当** 系统提取或应用 JM Cookie
- **那么** 原始 Cookie 只能在 Electron 主进程、登录窗口隔离上下文和本地 Python 子进程之间流转
- **且** React renderer 返回值、错误消息和诊断日志禁止包含 Cookie 值

### 需求:交互恢复必须有界并提供浏览器页面兜底

验证成功后系统必须自动重试原 Python 收藏夹请求一次；若仍为反爬挑战，系统必须优先使用同一已验证浏览器窗口捕获的合格页面快照交给 Python 解析。快照解析成功后，系统必须将 JM 收藏夹恢复策略切换为后续用户主动翻页优先静默快照恢复。禁止同一次用户动作再次自动弹窗或无限重试。

#### 场景:Cookie 同步后 Python 重试成功
- **当** 用户验证成功且第一次 Python 自动重试返回正常收藏夹页面
- **那么** 系统返回标准收藏夹结果并关闭验证流程
- **且** 禁止调用浏览器快照解析入口

#### 场景:Python 重试仍被挑战
- **当** 第一次 Python 自动重试仍返回结构化挑战且存在来自同一可信收藏夹 URL 的合格页面快照
- **那么** 系统将快照交给 Python JM 解析逻辑并返回解析后的收藏夹结果
- **且** 系统必须记录后续 JM 收藏夹用户翻页可优先使用静默快照恢复

#### 场景:快照不可用
- **当** Python 重试仍被挑战且快照缺失、超限、来源不可信或解析失败
- **那么** 系统停止自动恢复并返回带手动重试入口的可恢复错误
- **且** 一次用户动作不得触发第二次验证窗口

### 需求:取消验证不得破坏现有状态

用户取消或关闭验证窗口时，系统必须停止本次恢复，不得清除认证信息或把挑战报告为登录失效；收藏夹已有缓存必须保留。

#### 场景:有缓存时取消
- **当** 收藏夹已显示缓存内容且用户取消验证
- **那么** 页面继续显示缓存内容并提示刷新未完成
- **且** 禁止清空漫画、分页和下载状态缓存

#### 场景:无缓存时取消
- **当** 首次加载没有缓存且用户取消验证
- **那么** 页面显示可重试的人机验证提示
- **且** 禁止显示“登录凭证已失效”或自动再次弹窗

### 需求:验证后静默快照翻页恢复

当 JM 收藏夹交互恢复已通过可信浏览器页面快照成功解析结果后，系统必须允许后续用户主动翻页优先使用隐藏浏览器快照解析，避免先触发 Python 收藏夹直连请求。

#### 场景:快照兜底成功后进入静默模式
- **当** 用户通过可见 JM 验证窗口完成恢复，且 Python 自动重试仍被挑战但浏览器页面快照解析成功
- **那么** 系统必须记录静默快照恢复可用状态
- **且** 系统必须记录最近一次可信 JM 收藏夹快照 URL，供后续页码派生使用

#### 场景:后续翻页直接走隐藏快照
- **当** 用户主动加载 JM 收藏夹后续页，且静默快照恢复可用
- **那么** 主进程必须先使用隐藏 BrowserWindow 加载派生出的目标页收藏夹 URL 并捕获快照
- **且** 禁止在该静默尝试前调用 Python `get_favourites`

#### 场景:隐藏快照解析成功
- **当** 隐藏 BrowserWindow 捕获到来自可信 JM 收藏夹 URL 的合格页面快照
- **那么** 系统必须调用 Python 快照解析入口返回收藏夹结果
- **且** 禁止打开可见验证窗口

#### 场景:隐藏快照失败回退
- **当** 静默快照窗口超时、崩溃、加载到不可信 URL、仍处于强 challenge 页面或快照解析失败
- **那么** 系统必须回退到原有 Python `get_favourites` 和可见交互恢复流程
- **且** 禁止清除既有登录凭据和已显示收藏夹缓存

#### 场景:非交互请求不触发隐藏窗口
- **当** 缓存后台刷新、相邻页预加载或其他非交互调用请求 JM 收藏夹
- **那么** 系统禁止打开可见或隐藏 BrowserWindow
- **且** 调用必须按现有非交互错误处理路径结束

### 需求:验证窗口自动完成已渲染收藏夹页面

当可见 JM 验证窗口加载后已直接显示可信收藏夹页面时，系统必须自动完成快照捕获和认证上下文同步，避免要求用户点击“我已完成验证”。

#### 场景:加载后已经显示收藏夹
- **当** JM 验证窗口的当前 URL 是可信收藏夹 URL，且 DOM 中包含可解析的收藏夹内容
- **那么** 系统必须自动捕获页面快照并同步该窗口的 Cookie 与 User-Agent
- **且** 成功后必须按原验证完成流程关闭窗口并继续恢复收藏夹请求

#### 场景:仍处于挑战页
- **当** JM 验证窗口加载后仍包含强挑战标记且没有可解析收藏夹内容
- **那么** 系统必须保持窗口打开
- **且** 用户仍可手动完成站点验证或取消流程
