## 需求
### 需求:可靠识别 JM 反爬挑战响应

系统必须在处理 JM HTML 响应时识别 Cloudflare 反爬挑战。识别必须优先使用 `cf-mitigated: challenge` 等明确响应元数据，并辅以稳定的强挑战页标记；对于 `captcha` 等可能出现在正常页面脚本中的弱文本，系统必须结合页面是否存在可解析收藏夹内容进行判定。禁止以响应正文小于固定长度作为挑战成立的必要条件。

#### 场景:响应头明确标记挑战
- **当** JM 响应包含 `cf-mitigated: challenge`
- **那么** 系统必须将响应分类为反爬挑战，无论 HTTP 状态码和正文长度为何

#### 场景:长正文包含稳定挑战标记
- **当** JM 响应正文超过 500 字节且包含 Cloudflare 挑战平台或 “Just a moment” 等稳定标记
- **那么** 系统必须将响应分类为反爬挑战

#### 场景:正常长页面不被误判
- **当** JM 响应为正常 HTML 且不包含挑战响应头或稳定挑战页标记
- **那么** 系统禁止将其分类为反爬挑战

#### 场景:收藏夹内容优先于弱 challenge 文本
- **当** JM 收藏夹 HTML 中存在可解析的收藏夹条目或专辑链接
- **且** 页面仅包含 `captcha` 等弱 challenge 文本
- **那么** 系统禁止将其分类为反爬挑战

### 需求:收藏夹挑战执行有界恢复

JM 收藏夹请求遇到反爬挑战时，系统必须复用当前 Session、系统代理、Cookie、User-Agent 和目标域名进行有界后台恢复；重试次数必须有固定上限，禁止无限重试或创建绕过系统代理的新会话。后台恢复耗尽后必须产生包含受挑战 URL 的结构化挑战信号，是否启动用户交互由上层显式决定。

#### 场景:首次挑战后重试成功
- **当** 首次收藏夹请求被识别为反爬挑战，后续有界重试返回正常收藏夹页面
- **那么** 系统必须解析并返回收藏夹内容，且不得要求用户重新登录或启动验证窗口

#### 场景:挑战持续至重试耗尽
- **当** 所有允许的收藏夹后台重试仍被识别为反爬挑战
- **那么** 系统必须停止后台请求并返回专用挑战错误码、来源和实际受挑战 URL
- **且** 禁止继续无限重试、清空认证数据或直接在 Python 层启动 UI

#### 场景:恢复请求保持网络上下文
- **当** 系统执行收藏夹后台挑战恢复
- **那么** 所有恢复请求必须使用原 JM Session 及其已注入的系统代理和认证上下文

#### 场景:非交互调用耗尽挑战
- **当** 后台预加载或工具调用在挑战重试耗尽后未显式允许用户交互
- **那么** 系统必须以可恢复挑战错误结束
- **且** 禁止打开浏览器窗口

### 需求:反爬挑战与认证失效分开报告

系统必须使用显式错误类型和专用 IPC 错误码区分反爬挑战和认证失效。挑战错误的数据载荷必须仅包含经约束的来源、受挑战 URL 和安全消息；IPC 错误映射禁止仅因错误文本包含 `403`、`cloudflare` 或 `just a moment` 就把反爬挑战报告为登录凭证失效。

#### 场景:持续挑战的结构化提示
- **当** JM 收藏夹在后台重试后仍触发反爬挑战
- **那么** IPC 必须返回专用反爬挑战错误码和受挑战 URL
- **且** 消息表达“站点人机验证”语义，禁止返回“登录凭证已失效”

#### 场景:明确跳转登录页
- **当** JM 收藏夹请求最终跳转到登录页或正常页面明确要求登录
- **那么** 系统必须返回需要登录状态或认证错误
- **且** 禁止伪装成反爬挑战信号

#### 场景:普通 403 不伪装成挑战
- **当** JM 返回 403，但既无挑战响应头也无稳定挑战页标记
- **那么** 系统必须沿用普通 HTTP 或认证错误处理，禁止将其自动分类为反爬挑战

#### 场景:挑战载荷不泄露敏感信息
- **当** IPC 序列化反爬挑战错误
- **那么** `data` 禁止包含 Cookie、User-Agent 原文、HTML 正文或认证令牌

### 需求:浏览器快照挑战识别优先保留已渲染收藏夹内容

系统在判断 JM 收藏夹浏览器快照是否仍处于人机验证状态时，必须区分强挑战标记和弱挑战文本。若页面已经包含可解析收藏夹内容，系统禁止仅因 `captcha` 等弱文本存在就拒绝快照。

#### 场景:正常收藏夹页面含弱 captcha 文本
- **当** 浏览器快照 URL 为可信 JM 收藏夹 URL，且 HTML 包含收藏夹条目或专辑链接
- **且** HTML 中仅存在 `captcha` 等弱挑战文本
- **那么** 系统必须接受该快照并尝试解析收藏夹内容

#### 场景:强挑战标记仍被拒绝
- **当** 浏览器快照 HTML 包含 Cloudflare challenge platform、`cf-chl-`、`cf-challenge` 或 “Just a moment” 等强挑战标记
- **那么** 系统必须将该快照判定为仍处于反爬挑战状态
- **且** 禁止将该快照交给收藏夹条目解析逻辑作为成功结果

#### 场景:无收藏夹内容且含弱 captcha 文本
- **当** 浏览器快照不包含可解析收藏夹内容且包含 `captcha` 弱挑战文本
- **那么** 系统必须将该页面视为未完成验证或不可用快照

### 需求:JM 搜索请求必须检测反爬挑战并抛出结构化信号

JM 搜索请求（`JmParser.search`）遇到 Cloudflare 反爬挑战时，必须复用收藏夹同款检测逻辑（`_is_challenge_response` / `_is_challenge_page`）识别挑战，并抛出包含受挑战搜索 URL 的 `AntiBotChallengeError`。禁止将挑战页 HTML 静默解析为空结果列表返回；禁止在 `search()` 的异常兜底中吞掉 `AntiBotChallengeError`。

#### 场景:搜索响应头标记挑战

- **当** JM 搜索响应包含 `cf-mitigated: challenge` 响应头
- **那么** `search()` 必须抛出 `AntiBotChallengeError`，其 `challenge_url` 为本次搜索请求的 URL
- **且** 禁止返回空列表 `([], None)`

#### 场景:搜索正文包含稳定挑战标记

- **当** JM 搜索响应正文包含 Cloudflare 挑战平台或 "Just a moment" 等稳定挑战标记
- **那么** `search()` 必须抛出 `AntiBotChallengeError`，其 `challenge_url` 为本次搜索请求的 URL
- **且** 禁止将该正文交给 `_parse_search_results` 解析

#### 场景:正常搜索结果不受影响

- **当** JM 搜索返回正常 HTML（无挑战响应头、无稳定挑战页标记）
- **那么** `search()` 必须正常解析并返回搜索结果
- **且** 禁止因误判挑战而抛出 `AntiBotChallengeError`

#### 场景:搜索真无结果不伪装成挑战

- **当** JM 搜索返回正常 HTML 但确无匹配结果（解析后为空列表）
- **那么** `search()` 必须返回空列表 `([], None)`
- **且** 禁止将"无结果"误判为挑战

#### 场景:非挑战异常仍走兜底

- **当** JM 搜索因网络抖动等非挑战、非 `ParserResponseError` 异常失败
- **那么** `search()` 的 `except Exception` 兜底必须保持原行为，记录日志并返回 `([], None)`
- **且** `AntiBotChallengeError` 与 `ParserResponseError` 必须在兜底之前显式重新抛出，不被吞掉

### 需求:搜索 IPC handler 必须经 auth_error_guard 传播挑战错误

`handle_search` 中调用 `MultiSourceParser.search` 的主搜索分支必须使用 `_auth_error_guard` 上下文管理器，与 `handle_get_favourites` 对齐。`AntiBotChallengeError` 必须经 guard 冒泡到 `ipc_server.py` 顶层捕获，序列化为 JSON-RPC `-32002` 结构化信号 `{source: 'jm', challengeUrl, message}`。禁止在 `handle_search` 内捕获并吞掉 `AntiBotChallengeError`。

#### 场景:搜索挑战错误序列化为 -32002

- **当** JM 搜索抛出 `AntiBotChallengeError`
- **那么** `handle_search` 必须让其冒泡（经 `_auth_error_guard`）
- **且** `ipc_server.py` 顶层捕获后必须返回 JSON-RPC error code `-32002`，data 含 `source: 'jm'`、`challengeUrl`、`message`
- **且** 该错误可被 Electron 主进程 `isJmChallengeError` 精确识别

#### 场景:非 JM 来源搜索不受影响

- **当** hcomic / moeimg / bika / nh / copymanga 来源搜索请求
- **那么** 这些来源的 parser 不会抛出 `AntiBotChallengeError`（挑战检测仅在 JM parser 内）
- **且** `handle_search` 的行为对这些来源保持不变

### 需求:挑战恢复编排必须支持搜索请求复用

`jm-challenge-recovery.ts` 的核心恢复编排（打开挑战窗口 → cookie 同步 → 重试原请求一次）必须供收藏夹与搜索请求共用。搜索恢复必须把 Python 结构化错误中的原始受挑战 URL 交给挑战窗口；受信任的 JM 首页根 URL和 `/search/photos` URL 必须通过目标校验并实际打开，禁止被收藏夹专用路径规则误判为"人机验证地址无效"。搜索恢复必须复用 `apply_auth` 同步 Cookie 到全局 parser Session，并用原搜索参数重试 `bridge.call('search', ...)` 一次；禁止调用收藏夹快照兜底。

#### 场景:首页空搜索触发验证窗口
- **当** 用户主动执行 JM 空白关键词首页搜索，Python 返回 `challengeUrl=https://<可信域>/`
- **那么** 主进程必须打开 JM 挑战窗口加载该根 URL
- **且** 禁止返回"人机验证地址无效"或改写为收藏夹 URL

#### 场景:普通搜索触发验证窗口
- **当** 用户主动搜索 JM 且 Python 返回合法 `/search/photos?main_tag=0&search_query=...` 挑战 URL（`-32002`，`allowInteractiveChallenge=true`）
- **那么** 主进程必须打开 JM 挑战窗口加载原受挑战搜索 URL
- **且** 窗口叠层必须表达"人机验证"语义（复用 `mode='challenge'`）

#### 场景:非法搜索挑战地址仍被拒绝
- **当** 结构化挑战载荷包含非可信域、非 HTTPS、任意路径或白名单外搜索参数
- **那么** 主进程必须在开窗前拒绝目标并返回安全错误
- **且** 禁止为修复搜索恢复而关闭来源、路径或查询参数校验

#### 场景:验证后用原搜索参数重试
- **当** 用户在挑战窗口完成验证，Cookie 已同步到 Python parser Session
- **那么** 恢复编排必须用原始 `query / mode / page / source / tag` 参数调用 `bridge.call('search', ...)` 重试一次
- **且** 禁止递归进入恢复（重试再次被挑战时直接返回可重试错误）

#### 场景:搜索重试成功
- **当** 验证后第一次搜索重试返回正常结果
- **那么** 恢复编排必须返回该结果
- **且** 禁止调用任何收藏夹快照解析入口

#### 场景:搜索重试仍被挑战
- **当** 验证后第一次搜索重试仍返回结构化挑战错误
- **那么** 恢复编排必须停止并返回带手动重试入口的可恢复错误
- **且** 禁止调用收藏夹快照兜底（搜索无快照入口）
- **且** 一次用户动作不得触发第二次验证窗口

#### 场景:用户取消搜索验证
- **当** 用户取消或关闭搜索挑战窗口
- **那么** 恢复编排必须停止，返回 `cancelled=true` 的可恢复错误
- **且** 禁止清除认证信息或把挑战报告为登录失效

#### 场景:收藏夹恢复流程回归不变
- **当** 收藏夹请求触发 `recoverJmChallenge`（收藏专用入口）
- **那么** 其行为必须与变更前完全一致：开窗 → 重试 `get_favourites` → 必要时快照兜底 → 静默恢复状态管理
- **且** 收藏夹快照 URL 必须继续通过收藏夹专用严格校验

### 需求:搜索与收藏验证数据必须经共享 session 跨界面复用

JM 来源全进程只有一个 `JmParser` 实例和一个 `self.session`。无论用户在搜索页还是收藏页完成人机验证，提取的 Cookie + User-Agent 必须经 `apply_auth` → `configure_auth` → `_sync_cookies_to_jar` 注入该共享 session，使后续搜索请求和收藏夹请求都自动携带已验证 cookie，无需用户在另一界面重新验证。

#### 场景:搜索页验证后收藏页免重验

- **当** 用户在搜索页完成人机验证，cookie 已同步到全局 parser session
- **且** 用户随后切换到收藏页主动加载 JM 收藏夹
- **那么** 收藏夹请求必须自动携带已验证 cookie
- **且** 若 cookie 仍有效，收藏夹请求不得再次触发挑战窗口

#### 场景:收藏页验证后搜索页免重验

- **当** 用户在收藏页完成人机验证，cookie 已同步到全局 parser session
- **且** 用户随后在搜索页主动搜索 JM
- **那么** 搜索请求必须自动携带已验证 cookie
- **且** 若 cookie 仍有效，搜索不得再次触发挑战窗口
