# tag-favourites 规范

## 目的
定义用户主动收藏「推荐标签」（白名单 `my_tags`）的能力规范。覆盖 `my_tags` 的分来源隔离存储、`set_config` IPC 持久化、检测标签候选池挑选、手动输入、详情抽屉入口、与 `tag_blacklist` 的互斥校验，以及升级后的空启动行为。

## 需求

### 需求:用户可以主动收藏「推荐标签」并分来源隔离

系统必须支持用户主动将标签加入「推荐标签」白名单（`my_tags`），作为搜索结果卡片高亮的唯一生效源。`my_tags` 必须按来源严格隔离存储为 `dict[source] → list[str]` 结构，与 `tag_blacklist` 完全对称。同一标签字符串（如 `"NTR"`）在不同来源下必须独立管理，禁止跨来源共享或归一化。

支持的来源必须为 `hcomic`、`moeimg`、`jm`、`bika`，禁止包含 `copymanga`（其 parser 不解析标签）。

#### 场景:my_tags 按来源独立存储

- **当** 用户在 `jm` 来源下收藏标签 `"校園"`，在 `hcomic` 来源下收藏标签 `"人妻"`
- **那么** `my_tags` 必须存储为 `{"jm": ["校園"], "hcomic": ["人妻"], ...}`
- **且** 在 `hcomic` 来源下搜索时，`"校園"` 不得作为推荐标签生效

#### 场景:同一标签名在不同来源各自独立

- **当** `my_tags["jm"]` 含 `"NTR"`，`my_tags["nh"]` 不含 `"NTR"`
- **那么** 查看 jm 来源搜索结果时 `"NTR"` 是推荐标签，查看 nh 来源时不是
- **且** 在 nh 来源下收藏 `"NTR"` 不得影响 `my_tags["jm"]`

#### 场景:copymanga 来源不提供推荐标签能力

- **当** 当前来源为 `copymanga`
- **那么** 系统禁止允许用户为其添加推荐标签
- **且** 详情抽屉与设置页的推荐标签入口必须对该来源隐藏或禁用

#### 场景:大小写不敏感匹配与去重

- **当** `my_tags["jm"]` 已含 `"NTR"`，用户再次添加 `"ntr"`
- **那么** 系统必须去重，`my_tags["jm"]` 仍只含一个 `"NTR"`
- **当** 漫画标签为 `"Ntr"`，`my_tags["jm"]` 含 `"NTR"`
- **那么** 匹配判定必须为命中（大小写不敏感）

### 需求:推荐标签必须通过 set_config IPC 持久化，复用 tag_blacklist 的存储与校验模式

「推荐标签」(`my_tags`) 必须作为 `config.json` 的配置字段持久化，走现有 `get_config`/`set_config` IPC 通道，禁止新增独立的 Python IPC handler。其存储结构、来源键归一化、默认值补齐、原子写入规则必须与 `tag_blacklist` 完全一致。

#### 场景:通过 set_config 写入 my_tags

- **当** 前端通过 `set_config` 发送 `myTags` 字段
- **那么** Electron 主进程必须用对称 `tag_blacklist` 的校验规则校验（5 来源数组、每项非空字符串 ≤64 字符、去重、每源 ≤500 项）
- **且** 校验通过后必须原子写入 `config.json`

#### 场景:默认值与来源键补齐

- **当** `config.json` 不含 `my_tags` 键（旧版本文件）
- **那么** `Config.load()` 必须为全部 5 个来源键（含 `copymanga`）补齐空数组默认值
- **且** 不触发迁移或告警

#### 场景:get_config 返回 my_tags

- **当** 前端通过 `get_config` 读取配置
- **那么** 返回结果必须包含 `myTags`（camelCase），值为 `dict[source] → list[str]`，含全部 5 个来源键

### 需求:用户必须能从「检测标签」候选池挑选标签加入推荐标签

系统必须在设置页「推荐标签」区下方提供「检测标签」候选池展示区，列出 `sync_favourite_tags` 检测出的高频标签（来自 `favourite_tag_index`）。候选池仅供展示与挑选，禁止直接参与搜索结果高亮计算。用户点击候选 chip 必须将该标签加入当前来源的 `my_tags`。

#### 场景:候选池展示检测到的高频标签

- **当** 用户在设置页选择 `jm` 来源，且 `favourite_tag_index` 中 `jm` 来源有 `[{tag:"NTR", count:12}, {tag:"校園", count:8}]`
- **那么** 「检测标签」区必须展示 `"NTR (×12)"` 与 `"校園 (×8)"` 两个候选 chip
- **且** 这些标签不得直接使搜索结果高亮（必须先被用户挑选进 `my_tags`）

#### 场景:点击候选 chip 加入推荐标签

- **当** 用户点击候选池中的 `"NTR (×12)"` chip
- **那么** 系统必须将 `"NTR"` 加入 `my_tags["jm"]`
- **且** 该 chip 必须即时变为已选态（打勾置灰），表示已加入推荐标签

#### 场景:已选标签在候选池中打勾置灰

- **当** `my_tags["jm"]` 已含 `"NTR"`，且 `favourite_tag_index` 的 `jm` 来源也有 `"NTR"`
- **那么** 候选池中的 `"NTR"` chip 必须显示为已选态（打勾标记 + 置灰，不可重复点击加入）
- **且** 必须提供取消推荐的入口（如再次点击移除出 `my_tags`）

#### 场景:候选池为空时的引导态

- **当** 用户从未执行 `sync_favourite_tags`，`favourite_tag_index` 为空
- **那么** 「检测标签」区必须展示引导文案（如「请先同步收藏夹以生成检测标签」）
- **且** 必须提供「从收藏夹同步标签」按钮入口

### 需求:用户必须能手动输入任意标签加入推荐标签

设置页「推荐标签」区必须提供手动输入框，允许用户添加 `sync_favourite_tags` 未检测到的任意标签字符串到当前来源的 `my_tags`。输入必须遵循与候选池挑选相同的去重、大小写归一、长度限制规则。

#### 场景:手动输入新标签

- **当** 用户在 `hcomic` 来源的「推荐标签」区输入框键入 `"小众标签"` 并提交
- **那么** `"小众标签"` 必须被加入 `my_tags["hcomic"]`
- **且** 即使该标签不在 `favourite_tag_index` 中也必须接受

#### 场景:手动输入重复标签被去重

- **当** 用户手动输入一个 `my_tags["hcomic"]` 已含的标签（大小写不敏感）
- **那么** 系统必须拒绝重复添加，并给出可见提示

#### 场景:手动输入空字符串或超长字符串被拒绝

- **当** 用户提交空字符串、纯空白、或超过 64 字符的标签
- **那么** 系统必须拒绝并提示，禁止写入 `my_tags`

### 需求:用户必须能从详情抽屉的 tag chip 加入推荐标签

详情抽屉（`ComicInfoDrawer`）的 tag chip 小按钮点击后，必须弹出操作菜单，允许用户将当前标签加入当前来源的 `my_tags`（推荐）或 `tag_blacklist`（屏蔽）。弹窗选项必须根据标签当前状态动态展示，禁止平铺全部选项造成认知负担。

#### 场景:未设置的标签弹窗展示加入推荐与屏蔽

- **当** 用户点击一个既不在 `my_tags` 也不在 `tag_blacklist` 的标签的小按钮
- **那么** 弹窗必须展示「加入推荐标签」与「加入屏蔽标签」两个选项
- **且** 用户选择「加入推荐标签」后，该标签必须加入 `my_tags[当前来源]`

#### 场景:已推荐的标签弹窗展示取消推荐

- **当** 用户点击一个已在 `my_tags`（推荐态）的标签的小按钮
- **那么** 弹窗必须展示「取消推荐标签」选项（以及「加入屏蔽标签」选项，若需切换）
- **且** 用户选择「取消推荐标签」后，该标签必须从 `my_tags[当前来源]` 移除

#### 场景:已屏蔽的标签弹窗展示取消屏蔽

- **当** 用户点击一个已在 `tag_blacklist`（屏蔽态）的标签的小按钮
- **那么** 弹窗必须展示「取消屏蔽标签」选项
- **且** 用户选择「取消屏蔽」后，该标签必须从 `tag_blacklist[当前来源]` 移除

#### 场景:详情抽屉 tag chip 的推荐态视觉

- **当** 详情抽屉展示的某标签已在 `my_tags[当前来源]` 中（大小写不敏感）
- **那么** 该 tag chip 必须应用推荐态样式（与现有 `recommendedTagSet` 命中样式一致，琥珀色）
- **且** 推荐态与屏蔽态互斥，已屏蔽的标签不显示推荐态

### 需求:推荐标签与屏蔽标签互斥禁止冲突

同一来源下，一个标签禁止同时存在于 `my_tags` 与 `tag_blacklist`。当用户尝试将一个已屏蔽的标签加入推荐（或反之）时，系统必须拒绝并给出可见提示，禁止依赖优先级隐式裁决。

#### 场景:加入推荐时该标签已被屏蔽

- **当** 用户尝试将标签 `"NTR"` 加入 `my_tags["jm"]`，但 `"NTR"` 已在 `tag_blacklist["jm"]` 中
- **那么** 系统必须拒绝添加，并提示「该标签已被屏蔽，请先取消屏蔽」
- **且** 禁止将 `"NTR"` 写入 `my_tags["jm"]`

#### 场景:加入屏蔽时该标签已被推荐

- **当** 用户尝试将标签 `"人妻"` 加入 `tag_blacklist["jm"]`，但 `"人妻"` 已在 `my_tags["jm"]` 中
- **那么** 系统必须拒绝添加，并提示「该标签已是推荐标签，请先取消推荐」
- **且** 禁止将 `"人妻"` 写入 `tag_blacklist["jm"]`

### 需求:升级后 my_tags 必须空启动，禁止自动迁移存量数据

系统升级引入 `my_tags` 时，禁止自动将现有 `favourite_tag_index` 的存量数据迁移为 `my_tags`。`my_tags` 初始必须为空，用户必须手动通过候选池挑选、手动输入或详情抽屉添加来填充。决策权必须完全交还用户。

#### 场景:升级后 my_tags 为空

- **当** 用户从旧版本升级到引入 `my_tags` 的版本
- **那么** 首次启动时 `my_tags` 必须为所有来源的空数组
- **且** 禁止自动从 `favourite_tag_index` 复制任何标签到 `my_tags`

#### 场景:升级后高亮暂时失效并引导用户

- **当** 用户升级后首次进入搜索页，且 `my_tags` 为空
- **那么** 搜索结果不得显示任何推荐高亮
- **且** 设置页「检测标签」候选池必须仍展示存量 `favourite_tag_index` 数据，引导用户挑选
