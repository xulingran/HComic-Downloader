## 新增需求

### 需求:重复组指纹计算

系统必须为每个检测出的重复组计算一个稳定的代表指纹，指纹定义为组内所有漫画标题经 `normalizeTitle` 处理后的字典序最小值。指纹必须与收藏分页返回顺序、组成员增删无关（只要字典序最小的成员仍在组内）。指纹禁止拼接来源前缀。

#### 场景:指纹取字典序最小归一化标题
- **当** 一个重复组包含标题归一化后为 "apple"、"banana"、"cherry" 的三本漫画
- **那么** 该组的指纹为 "apple"

#### 场景:指纹与成员输入顺序无关
- **当** 同一组漫画以不同的数组顺序传入检测（如因收藏分页顺序变化）
- **那么** 计算出的组指纹必须相同

#### 场景:非最小成员变更不影响指纹
- **当** 组内字典序最小的成员仍在组中，而其他成员被取消收藏或被站方改名
- **那么** 该组的指纹必须保持不变

### 需求:黑名单条目结构

系统必须将已忽略的重复组以结构化条目持久化到配置字段 `duplicate_blacklist`，结构为 `dict[str, list[dict]]`，其中外层 key 为来源标识（hcomic / moeimg / jmcomic / bika），内层列表的每项为对象 `{"fingerprint": str, "memberCount": int | null}`。`fingerprint` 为组指纹（`normalizeTitle` 结果），`memberCount` 为该组被忽略时的成员数量（首次忽略时记录实际值；从旧版纯字符串数据迁移而来的条目为 null，表示基线未知）。黑名单必须按来源隔离，禁止跨来源共享。该字段必须通过 `CONFIG_KEY_MAP` 与前端 camelCase 键 `duplicateBlacklist` 双向映射，并在 `AppConfig` / `ConfigKey` / `CONFIG_KEYS` 三处保持一致。

#### 场景:新字段默认值为空
- **当** 加载一个不含 `duplicate_blacklist` 字段的老配置文件
- **那么** 系统必须将该字段填充为空字典（每个来源对应空列表），且不报错

#### 场景:旧版纯字符串数据静默迁移
- **当** 加载到的 `duplicate_blacklist` 条目为纯字符串（旧版格式）
- **那么** 系统必须将其转换为 `{"fingerprint": <该字符串>, "memberCount": null}`，不报错且不丢失指纹

#### 场景:按来源隔离存储
- **当** 用户在 hcomic 来源下忽略指纹 "abc"
- **那么** 该条目必须仅出现在 `duplicate_blacklist["hcomic"]` 中，禁止出现在其他来源的列表中

### 需求:忽略此组操作

用户必须能够将一个 active 重复组标记为已忽略。点击"忽略此组"后，系统必须向当前来源的黑名单追加条目 `{fingerprint: 组指纹, memberCount: group.comics.length}`，该组立即转为 ignored 态并以折叠形式渲染。同一指纹禁止在列表中重复出现（已存在时更新其 memberCount 为当前组大小）。

#### 场景:标记忽略时记录成员数
- **当** 用户在一个包含 3 本漫画的 active 组上点击"忽略此组"
- **那么** 黑名单新增条目 `{fingerprint: <组指纹>, memberCount: 3}`，该组转为折叠态

#### 场景:重复忽略同一指纹时更新成员数
- **当** 用户对指纹已存在于黑名单的组再次触发忽略
- **那么** 黑名单中该指纹的条目必须仍只出现一次，且其 memberCount 更新为当前组的成员数

### 需求:取消忽略操作

用户必须能够将一个 ignored 重复组恢复为 active。取消忽略的途径必须至少包括：(1) ignored 组头部的"取消忽略"按钮；(2) 管理面板中的逐项移除。两种途径都禁止要求二次确认（取消忽略是非破坏性操作）。取消忽略后，该指纹条目必须从当前来源的黑名单列表移除，该组立即转为 active 态并以展开形式渲染。

#### 场景:从结果区取消忽略
- **当** 用户在某 ignored 组上点击"取消忽略"
- **那么** 该指纹条目从黑名单移除，该组立即转为 active 态并展开

#### 场景:从管理面板取消忽略
- **当** 用户在管理面板中点击某条目的移除按钮
- **那么** 该条目从对应来源的黑名单移除，且不弹出二次确认对话框

### 需求:检测结果分层渲染

检测完成后，系统必须按组指纹是否在当前来源的黑名单中，将重复组拆分为 active 和 ignored 两段进行渲染。active 组必须默认展开渲染，每组头部右侧带"忽略此组"按钮。ignored 组必须默认折叠渲染，每组头部右侧带"取消忽略"按钮。active 区段必须渲染在 ignored 区段之上。所有组必须始终被检测和渲染，禁止因黑名单中存在指纹而过滤掉任何组。

#### 场景:active 组在上 ignored 组在下
- **当** 检测发现 5 个组，其中 2 个指纹在黑名单中
- **那么** 3 个 active 组渲染在上方，2 个 ignored 组渲染在下方，中间以"已忽略"分隔标题区隔

#### 场景:ignored 组默认折叠
- **当** 一个组的指纹在当前来源的黑名单中
- **那么** 该组首次渲染时必须处于折叠态

### 需求:检测统计文案

检测完成且 `totalFetched > 0` 时，统计文案必须同时体现总组数与已忽略组数。当存在已忽略组时，文案格式必须为 `发现 N 组疑似重复（其中 M 组已忽略）`；当不存在已忽略组时，文案必须为 `发现 N 组疑似重复`。

#### 场景:存在已忽略组
- **当** 检测发现 5 个组，其中 2 个被忽略
- **那么** 统计文案显示为"已分析 X 本漫画，发现 5 组疑似重复（其中 2 组已忽略）"

#### 场景:不存在已忽略组
- **当** 检测发现 5 个组，其中 0 个被忽略
- **那么** 统计文案显示为"已分析 X 本漫画，发现 5 组疑似重复"

### 需求:成员变动检测与徽章提示

系统必须检测已忽略组在本次检测中的成员数量是否与记录的基线（memberCount）不一致，并在"管理已忽略"按钮上以数字徽章形式提示当前来源下存在成员变动的条目数量。memberCount 为 null 的条目（旧数据迁移而来）在本次检测中找到对应组时，必须静默填充 memberCount 为当前组成员数，且不计入变动徽章。memberCount 非 null 且与当前组成员数不等的条目必须计入徽章数字。徽章数字仅统计当前选中来源下的变动条目。当无变动条目时，徽章禁止显示。

#### 场景:基线建立后成员变化触发徽章
- **当** 一个已忽略条目 memberCount=3，本次检测到对应组实际有 4 本漫画
- **那么** 该条目计入徽章，"管理已忽略"按钮显示数字 +1

#### 场景:旧数据 null 静默填充不触发徽章
- **当** 一个已忽略条目 memberCount=null（旧版迁移），本次检测到对应组有 3 本漫画
- **那么** 该条目的 memberCount 被静默更新为 3，且不计入本次徽章数字

#### 场景:组消失不触发徽章
- **当** 一个已忽略条目的指纹在本次检测结果中找不到对应组（用户已取消收藏或标题变化）
- **那么** 该条目不计入徽章（系统无法判断成员数，保持原状）

#### 场景:无变动时不显示徽章
- **当** 当前来源下所有已忽略条目的 memberCount 都等于对应组的实际成员数（或为 null）
- **那么** "管理已忽略"按钮上禁止显示徽章

### 需求:管理面板成员变动展示与确认

管理面板必须对存在成员变动的条目做出视觉区分（如标记或高亮），并为每个变动条目提供"确认"操作。点击"确认"必须将该条目的 memberCount 更新为当前检测到的组成员数（清除变动状态）。变动条目同样支持"取消忽略"移除。

#### 场景:变动条目视觉标记
- **当** 管理面板中某条目 memberCount 与最近一次检测到的组成员数不等
- **那么** 该条目必须有明显的视觉标记（如变色、图标、文案提示成员数变化）

#### 场景:确认操作更新基线
- **当** 用户在某变动条目上点击"确认"
- **那么** 该条目的 memberCount 更新为当前检测到的组成员数，变动标记消除

#### 场景:确认后徽章数字减少
- **当** 用户在管理面板确认了一个变动条目
- **那么** "管理已忽略"按钮上的徽章数字相应减少（若减至 0 则徽章消失）

### 需求:管理已忽略面板

工具箱·重复检测区块必须提供"管理已忽略"入口（位于区块标题行），点击后弹出面板。面板必须支持按来源切换、列出当前来源下所有已忽略条目（显示指纹及成员变动状态）、逐项"确认"（仅变动条目）或"取消忽略"。面板必须遵循项目既有的弹窗交互范式（点击背景遮罩关闭）。面板内移除操作禁止要求二次确认。当某来源下无已忽略条目时，必须显示空态文案。

#### 场景:打开管理面板
- **当** 用户点击"管理已忽略"按钮
- **那么** 弹出面板，默认展示当前选中的来源（与检测来源一致或首个来源）的已忽略条目列表

#### 场景:切换来源
- **当** 用户在面板内点击不同来源的 tab
- **那么** 列表必须更新为该来源下的已忽略条目

#### 场景:空来源显示空态
- **当** 当前选中的来源下没有已忽略条目
- **那么** 面板必须显示空态文案（如"暂无已忽略的重复组"）

#### 场景:点击背景遮罩关闭面板
- **当** 用户点击面板外的遮罩区域
- **那么** 面板必须关闭，且不丢失任何已做的确认/移除操作（操作已在点击时即时持久化）
