# download-dir-change-migration 规范

## 目的
变更下载目录时自动迁移旧目录下被历史记录引用的文件并同步更新 output_path，使下载目录变更成为"文件移动 + 数据库更新 + 配置落库"的原子操作，避免历史记录与新目录脱节导致健康检查误报 missing_file。

## 需求
### 需求:变更下载目录时自动迁移已记录文件

当用户通过设置页变更「下载目录」且新旧目录不同时，系统必须自动将旧目录下被 `download_history` 记录的文件迁移到新目录，并同步更新每条记录的 `output_path`，使下载目录变更成为"文件移动 + 数据库更新 + 配置落库"的原子操作。迁移必须复用既有 `migration-engine` 的 full migration 能力，禁止重复实现文件移动逻辑。

#### 场景:旧目录无可迁移记录时走快速路径

- **当** 用户改下载目录，且旧目录下没有任何被 `download_history` 记录且仍存在的文件（`plan_full_migration` 返回 `total_items == 0`）
- **那么** 系统直接落库新 `download_dir` 并更新运行时输出目录，不启动迁移线程，前端无迁移感知

#### 场景:检测到可迁移文件时前端必须知情确认

- **当** 用户改下载目录，且 `plan_full_migration` 检测到 N（N > 0）个需迁移的文件
- **那么** 前端必须弹出确认对话框，明确告知"将迁移 N 个文件到新目录并更新历史记录"，用户确认后才执行迁移；用户取消则不改动目录与配置

#### 场景:迁移异步执行且配置落库延后

- **当** 迁移启动
- **那么** 系统在后台线程执行文件移动（复用 `migration-engine` 的 `_run_migration` 线程模型），主线程立即返回迁移 ID 供前端展示进度；新 `download_dir` 必须在迁移完成回调中才落库，迁移进行期间保持旧配置

#### 场景:迁移期间暂停下载任务

- **当** 下载目录迁移启动
- **那么** 系统必须先暂停下载管理器（`toggle_global_pause`），防止迁移与下载同时写文件；迁移完成后恢复下载管理器

#### 场景:迁移全部成功后落库新目录

- **当** 迁移完成且 `state.status == "completed"`（无失败项）
- **那么** 系统落库新 `download_dir` 并应用运行时输出目录

#### 场景:迁移部分成功仍落库新目录并提示失败项

- **当** 迁移完成但有部分失败项（`state.failed_items` 非空），且至少有一个文件成功移动
- **那么** 系统仍落库新 `download_dir`（已移动文件已在新目录，不落库会造成新的不一致），前端必须显著提示"M 个文件迁移失败，请查看迁移日志"

#### 场景:迁移取消或全部失败时保持旧配置

- **当** 迁移被用户取消（`state.status == "cancelled"`）或全部文件移动失败
- **那么** 系统必须不落库新 `download_dir`，保持旧配置不变（文件位置未变，旧配置仍正确）

#### 场景:迁移进度复用既有通知通道

- **当** 下载目录迁移进行中
- **那么** 系统必须通过既有的 `migration_progress` JSON-RPC 通知推送进度，前端复用既有迁移进度 UI，禁止新增专用通道

#### 场景:重复改目录以当前状态为准

- **当** 用户多次改下载目录（含上次迁移部分失败后再次变更）
- **那么** 每次变更都基于当前 DB 与文件实际位置重新 `plan_full_migration` 评估，已在新目录的文件自然跳过，不重复移动
