# cache-directory-access 规范

## 目的
向用户暴露缓存文件（封面缓存、预览缓存）所在的磁盘目录：在设置页面「缓存管理」区域展示该目录的绝对路径，并提供「在系统文件管理器中打开」的能力，便于用户定位、排查与备份缓存。
## 需求
### 需求:设置页必须展示缓存文件所在目录路径

系统必须在设置页面「缓存管理」区域向用户展示缓存文件（封面缓存数据库、预览缓存数据库及预览图片文件）所在根目录的绝对路径。该路径必须由后端权威返回，禁止由前端硬编码目录名推导。

#### 场景:正常加载缓存管理区域
- **当** 用户进入设置页面的「缓存管理」区域
- **那么** 该区域顶部显示一行「缓存目录」，内容为缓存根目录的绝对路径（如 Windows 下的 `C:\Users\<user>\.hcomic_downloader`，POSIX 下的 `/home/<user>/.hcomic_downloader`）

#### 场景:后端返回路径已规范化为绝对路径
- **当** 前端调用 `getCacheDir()` 取回缓存目录
- **那么** 返回值的 `dir` 字段必须是一个非空的、经 `os.path.abspath` 规范化的绝对路径字符串

#### 场景:获取缓存目录失败时降级展示
- **当** `getCacheDir()` 调用失败或返回无效路径
- **那么** 「缓存目录」行显示「无法获取缓存目录」，且「打开目录」按钮被禁用，同时禁止影响该区域其余功能（缓存统计、上限滑块、清除按钮）的正常使用

### 需求:用户必须能从设置页打开缓存目录

系统必须在「缓存目录」行提供「打开目录」按钮，点击后调用系统文件管理器并定位到缓存根目录。该动作必须复用与「打开下载目录」一致的安全校验：路径必须为绝对路径、必须不含路径遍历片段（`..`）、必须不含控制字符、且必须是已存在的目录；任一校验失败时禁止打开并向上层抛错。

#### 场景:点击打开目录成功
- **当** 用户点击「打开目录」按钮且缓存根目录存在且可访问
- **那么** 系统调用系统文件管理器打开该目录，并返回 `{ success: true }`

#### 场景:缓存目录不存在时打开失败
- **当** 用户点击「打开目录」按钮，但缓存根目录在文件系统上不存在（如被外部删除）
- **那么** 系统禁止打开文件管理器，向前端抛出包含原因的错误，前端以 toast/提示形式告知用户「无法打开目录」

#### 场景:Preload 早期拒绝非法路径
- **当** 渲染进程传入非字符串、空字符串、超长（>1024）、非绝对路径、含 `..` 遍历或含控制字符的路径给 `openCacheDir`
- **那么** preload 必须在进入 IPC 之前即抛错，主进程不被调用

### 需求:缓存目录路径必须来源于缓存实例的真实位置

后端 `get_cache_dir` 返回的路径必须基于已初始化的封面/预览缓存实例所使用的真实目录（封面缓存数据库文件所在目录），而非在请求处理器中重复硬编码默认目录常量；以保证注入自定义目录的单元测试能取回一致路径，且未来缓存目录可配置时该方法自动跟随。

#### 场景:测试注入自定义目录后取回一致
- **当** 单元测试以自定义 `db_path` 构造缓存实例并请求缓存目录
- **那么** `get_cache_dir` 返回的路径必须等于该自定义目录（经规范化），而非默认的 `~/.hcomic_downloader`

### 需求:自定义协议 handler 必须复用缓存目录路径安全语义

`app-image://` 协议 handler 在解析 `url_hash` 到磁盘文件路径时，**必须**复用本能力既有的路径安全语义：最终路径必须为绝对路径、必须不含路径遍历片段（`..`）、必须约束在授权的缓存目录（`CoverCacheDB.files_dir` 或 `PreviewCacheDB.files_dir`）内。这通过 `path.resolve` + 前缀检查实现，与「打开下载/缓存目录」的安全校验同源。

handler 的缓存目录来源**必须**基于已初始化的缓存实例的真实 `files_dir`（遵循「缓存目录路径必须来源于缓存实例的真实位置」需求），**禁止**在 handler 内重复硬编码默认目录常量，以保证自定义目录的单元测试能一致覆盖。

#### 场景:协议 handler 使用缓存实例真实 files_dir

- **当** 协议 handler 解析 `app-image://cover/{url_hash}`
- **那么** 定位到的目录必须等于已初始化的 `CoverCacheDB.files_dir`
- **且** 自定义 `files_dir` 的测试环境下 handler 跟随该自定义目录

#### 场景:协议 handler 复用路径遍历校验

- **当** 请求路径经 `path.resolve` 后逃出缓存 `files_dir`
- **那么** handler 拒绝（返回 4xx），与「打开目录」对 `..` 遍历的拒绝语义一致

#### 场景:cover 与 preview 路由到各自真实目录

- **当** 协议 URL 首段为 `cover` 或 `preview`
- **那么** handler 分别使用 `CoverCacheDB.files_dir` 或 `PreviewCacheDB.files_dir` 的真实路径
- **且** 两者可能不同（虽默认同根，但遵循各自实例的真实位置）
