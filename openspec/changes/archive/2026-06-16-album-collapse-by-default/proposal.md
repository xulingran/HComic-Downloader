## 为什么

下载管理界面中，多章专辑会一次性展开所有章节子行，每章都占据一行独立的卡片。当专辑章数较多（如 8 章、10 章）或同时存在多个专辑时，下载列表会被拉得很长，用户难以纵览整体进度，需要滚动才能找到目标专辑。用户希望默认只看到"整个专辑"这一层级的进度条与控制项，章节明细折叠起来按需查看。

## 变更内容

- 专辑卡新增"折叠 / 展开"状态：默认折叠章节子行，只展示专辑标题、完成进度、专辑级进度条与专辑级控制按钮（暂停 / 恢复 / 重试 / 取消 / 强制打包）。
- 专辑存在失败章节时**自动展开**，避免用户错过需要重试的章节。
- 展开后章节子行按"失败优先"排序，失败章节始终排在最前。
- 头部非按钮区域与折叠箭头均可切换折叠；专辑级按钮（全部暂停 / 全部恢复 / 重试失败 / 全部取消 / 强制打包）点击不触发折叠切换。
- 用户手动展开/折叠后，状态在当前会话内保持；失败恢复为正常后不自动折叠，尊重用户操作。
- 折叠状态不持久化，刷新页面后重新按"有失败则展开、否则折叠"的默认规则初始化。
- 独立任务卡（非专辑或单章任务）行为保持不变。

## 功能 (Capabilities)

- `download-album-collapse`: 下载管理界面多章专辑的章节折叠展示行为，定义默认折叠、失败自动展开、失败优先排序、用户操作优先等可观测需求。

## 影响

- 受影响代码：`src/pages/DownloadPage.tsx`（专辑卡渲染逻辑，新增折叠 state、排序、点击区域处理）。
- 不影响：IPC 通道、Python 后端、Zustand store 结构、shared types、`ProgressBar` 组件接口、独立任务卡渲染。
- 不影响数据持久化：折叠状态纯前端、仅会话内有效。
- 不破坏现有功能：专辑级 / 章节级控制（暂停 / 恢复 / 重试 / 取消 / 强制打包）的全部交互行为不变，仅改变可见性。
