## 上下文

当前 `src/pages/DownloadPage.tsx` 中，专辑卡（`albumGroups` 渲染分支，约 209–360 行）每次都把该专辑下所有章节作为独立子行渲染（300–357 行）。每个章节子行包含独立的标题、状态徽标、章节级控制按钮（暂停 / 恢复 / 取消 / 重试）和进度条。

随着章节数增多或并发专辑数增多，列表纵向高度急剧膨胀，专辑级进度条淹没在章节行里，难以纵览。

本次变更只在前端单文件 `DownloadPage.tsx` 内引入"章节折叠"行为。store 结构、IPC、`ProgressBar` 组件、独立任务卡渲染均不变。

## 目标 / 非目标

**目标：**
- 专辑默认折叠章节子行，只展示专辑级信息与控制项。
- 失败章节始终对用户可见（自动展开 + 排序置顶），不漏掉需要用户介入的状态。
- 用户对折叠状态的操作（展开 / 折叠）在会话内被尊重，不因状态变化自动反转变。
- 专辑级按钮和折叠切换互不干扰。
- 不引入新依赖、不改变数据流、不持久化折叠状态。

**非目标：**
- 不做折叠状态的跨会话持久化（不写 localStorage / config）。
- 不改变章节级控制的语义或 IPC（暂停 / 恢复 / 取消 / 重试行为不变）。
- 不重构专辑分组逻辑（`albumGroups` / `albumTaskIds` 计算保持原样）。
- 不调整独立任务卡的展示。
- 不做"全部展开 / 全部折叠"批量操作（如后续有需求再加）。

## 决策

### D1：折叠状态存放在组件本地 state，键为 album group key

采用 `useState<Set<string>>`，存储**已展开**的 key 集合（即"展开白名单"）。
- key 复用现有 `${sourceSite}_${albumId}`（DownloadPage.tsx:49）。
- 选 "展开集合" 而非 "折叠集合"：默认态（集合为空时）配合"失败自动展开"逻辑更直观——失败专辑在初始化 / 状态变化时把 key 加入集合，其余默认不在集合中即折叠。

**替代方案**：`Record<string, boolean>`。否决——`Set` 语义更清晰，成员判定 O(1)，增删简单。

### D2：默认折叠 + 失败自动展开，用两个独立机制

- **初始化（首渲染 / 新专辑出现）**：专辑卡渲染时，若该 key 不在 state 中，且 `hasAnyFailed` 为 true → 视为展开；否则折叠。即"懒默认值"。
- **失败上升触发展开**：用 `useEffect` 监听每个 group 的 `hasAnyFailed`。当从 `false` 变为 `true` 时，把 key 加入展开集合。
- **用户操作优先**：用户手动折叠时从集合移除 key，手动展开时加入。一旦用户操作过，后续失败上升不再自动展开（因为已经在集合中或刚被用户移除——简化为"任何状态变化都不覆盖用户最近一次显式操作"）。

实现上，为简化判断，采用**双状态**：
- `expandedAlbums: Set<string>` —— 显式展开 / 折叠的目标状态。
- 用户点击切换时直接增删这个集合，不再依赖"懒默认值"。
- 初始挂载：对每个 group 计算初始值——有失败加入集合，无失败不加入。
- 失败上升：监听 `hasAnyFailed` 变化，仅当从 false→true 且 key 不在集合中时加入。**不**在失败恢复时自动移除（避免用户操作时列表抖动）。

**替代方案 A**：完全用"懒默认值 + 派生"，不存显式状态。否决——用户切换后的状态需要持久（会话内），派生无法表达"用户最近操作"。
**替代方案 B**：把失败时强制展开做成不可折叠。否决——会让用户失去对失败专辑折叠的能力，违背"尊重用户操作"。

### D3：头部点击区与按钮隔离

专辑卡头部 `div` 绑定 `onClick={toggle}`。所有专辑级按钮（全部暂停 / 全部恢复 / 重试失败 / 全部取消 / 强制打包）的 `onClick` 内调用 `e.stopPropagation()`，阻止冒泡到头部。折叠箭头作为头部内一个视觉元素，本身不需要单独 handler（点击头部任意非按钮位置都切换）。

**替代方案**：只让箭头按钮可点击。否决——点击面积小、不顺手。

### D4：展开时章节按"失败优先、其余保持原序"排序

用 `useMemo` 对 `group.tasks` 做稳定排序：失败章节（`status === 'failed'`）置顶，其余按原数组顺序。不引入二级排序键，避免章节顺序被意外打乱（章节本身有顺序语义）。

### D5：折叠时不渲染章节子行 DOM

折叠态下 `group.tasks.map(...)` 整段不渲染，改为单行提示：`▶ 展开查看 N 章`（N = `group.tasks.length`）。展开时渲染子行 + 头部箭头改为 `▼`、提示文案改为 `▼ 隐藏章节`。

**替代方案**：用 CSS `display:none` 保留 DOM。否决——章节数多时 DOM 节点仍然存在，浪费渲染；本项目无动画需求，条件渲染更简单。

## 风险 / 权衡

- **[用户错过失败章节]** → 缓解：失败上升时自动展开 + 展开时失败置顶 + 专辑卡头部已有 `(有失败)` 文案（DownloadPage.tsx:291）。三重信号确保可见性。
- **[用户手动折叠后再次失败，专辑不自动展开]** → 这是 D2 的有意权衡（尊重用户操作）。缓解：头部 `(有失败)` 红字 + 专辑级"重试失败"按钮在折叠态仍可见，用户可点击重试或手动展开。
- **[状态过滤（"失败" / "进行中"）下折叠专辑看不到具体失败章]** → 暂不处理：用户选了"失败"过滤器仍能看到红字 `(有失败)` 与"重试失败"按钮，需要细看时手动展开。若反馈不佳，后续可在"失败"过滤器下默认展开。
- **[多专辑并发时列表仍可能很长]** → 本次变更直接缓解（每专辑压缩为一行）。不做"全部折叠"批量按钮，按需再加。
- **[刷新后折叠状态丢失]** → 非目标。默认规则（有失败展开）能在大多数情况下给出合理初始态。
