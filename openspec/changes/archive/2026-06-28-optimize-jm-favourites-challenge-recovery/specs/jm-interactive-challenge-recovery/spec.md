## 新增需求

### 需求:验证后静默快照翻页恢复

当 JM 收藏夹交互恢复已通过可信浏览器页面快照成功解析结果后，系统必须允许后续用户主动翻页优先使用隐藏浏览器快照解析，避免先触发 Python 收藏夹直连请求。

#### 场景:快照兜底成功后进入静默模式
- **当** 用户通过可见 JM 验证窗口完成恢复，且 Python 自动重试仍被挑战但浏览器页面快照解析成功
- **那么** 系统必须记录静默快照恢复可用状态
- **且** 系统必须记录最近一次可信 JM 收藏夹快照 URL，供后续页码派生使用

#### 场景:后续翻页直接走隐藏快照
- **当** 用户主动加载 JM 收藏夹后续页，且静默快照恢复可用
- **那么** 主进程必须先使用隐藏 BrowserWindow 加载派生出的目标页收藏夹 URL 并捕获快照
- **且** 禁止在该静默尝试前调用 Python `get_favourites`

#### 场景:隐藏快照解析成功
- **当** 隐藏 BrowserWindow 捕获到来自可信 JM 收藏夹 URL 的合格页面快照
- **那么** 系统必须调用 Python 快照解析入口返回收藏夹结果
- **且** 禁止打开可见验证窗口

#### 场景:隐藏快照失败回退
- **当** 静默快照窗口超时、崩溃、加载到不可信 URL、仍处于强 challenge 页面或快照解析失败
- **那么** 系统必须回退到原有 Python `get_favourites` 和可见交互恢复流程
- **且** 禁止清除既有登录凭据和已显示收藏夹缓存

#### 场景:非交互请求不触发隐藏窗口
- **当** 缓存后台刷新、相邻页预加载或其他非交互调用请求 JM 收藏夹
- **那么** 系统禁止打开可见或隐藏 BrowserWindow
- **且** 调用必须按现有非交互错误处理路径结束

### 需求:验证窗口自动完成已渲染收藏夹页面

当可见 JM 验证窗口加载后已直接显示可信收藏夹页面时，系统必须自动完成快照捕获和认证上下文同步，避免要求用户点击“我已完成验证”。

#### 场景:加载后已经显示收藏夹
- **当** JM 验证窗口的当前 URL 是可信收藏夹 URL，且 DOM 中包含可解析的收藏夹内容
- **那么** 系统必须自动捕获页面快照并同步该窗口的 Cookie 与 User-Agent
- **且** 成功后必须按原验证完成流程关闭窗口并继续恢复收藏夹请求

#### 场景:仍处于挑战页
- **当** JM 验证窗口加载后仍包含强挑战标记且没有可解析收藏夹内容
- **那么** 系统必须保持窗口打开
- **且** 用户仍可手动完成站点验证或取消流程

## 修改需求

### 需求:交互恢复必须有界并提供浏览器页面兜底

验证成功后系统必须自动重试原 Python 收藏夹请求一次；若仍为反爬挑战，系统必须优先使用同一已验证浏览器窗口捕获的合格页面快照交给 Python 解析。快照解析成功后，系统必须将 JM 收藏夹恢复策略切换为后续用户主动翻页优先静默快照恢复。禁止同一次用户动作再次自动弹窗或无限重试。

#### 场景:Cookie 同步后 Python 重试成功
- **当** 用户验证成功且第一次 Python 自动重试返回正常收藏夹页面
- **那么** 系统返回标准收藏夹结果并关闭验证流程
- **且** 禁止调用浏览器快照解析入口

#### 场景:Python 重试仍被挑战
- **当** 第一次 Python 自动重试仍返回结构化挑战且存在来自同一可信收藏夹 URL 的合格页面快照
- **那么** 系统将快照交给 Python JM 解析逻辑并返回解析后的收藏夹结果
- **且** 系统必须记录后续 JM 收藏夹用户翻页可优先使用静默快照恢复

#### 场景:快照不可用
- **当** Python 重试仍被挑战且快照缺失、超限、来源不可信或解析失败
- **那么** 系统停止自动恢复并返回带手动重试入口的可恢复错误
- **且** 一次用户动作不得触发第二次验证窗口
