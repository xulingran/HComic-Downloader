## 新增需求

### 需求:浏览器快照挑战识别优先保留已渲染收藏夹内容

系统在判断 JM 收藏夹浏览器快照是否仍处于人机验证状态时，必须区分强挑战标记和弱挑战文本。若页面已经包含可解析收藏夹内容，系统禁止仅因 `captcha` 等弱文本存在就拒绝快照。

#### 场景:正常收藏夹页面含弱 captcha 文本
- **当** 浏览器快照 URL 为可信 JM 收藏夹 URL，且 HTML 包含收藏夹条目或专辑链接
- **且** HTML 中仅存在 `captcha` 等弱挑战文本
- **那么** 系统必须接受该快照并尝试解析收藏夹内容

#### 场景:强挑战标记仍被拒绝
- **当** 浏览器快照 HTML 包含 Cloudflare challenge platform、`cf-chl-`、`cf-challenge` 或 “Just a moment” 等强挑战标记
- **那么** 系统必须将该快照判定为仍处于反爬挑战状态
- **且** 禁止将该快照交给收藏夹条目解析逻辑作为成功结果

#### 场景:无收藏夹内容且含弱 captcha 文本
- **当** 浏览器快照不包含可解析收藏夹内容且包含 `captcha` 弱挑战文本
- **那么** 系统必须将该页面视为未完成验证或不可用快照

## 修改需求

### 需求:可靠识别 JM 反爬挑战响应

系统必须在处理 JM HTML 响应时识别 Cloudflare 反爬挑战。识别必须优先使用 `cf-mitigated: challenge` 等明确响应元数据，并辅以稳定的强挑战页标记；对于 `captcha` 等可能出现在正常页面脚本中的弱文本，系统必须结合页面是否存在可解析收藏夹内容进行判定。禁止以响应正文小于固定长度作为挑战成立的必要条件。

#### 场景:响应头明确标记挑战
- **当** JM 响应包含 `cf-mitigated: challenge`
- **那么** 系统必须将响应分类为反爬挑战，无论 HTTP 状态码和正文长度为何

#### 场景:长正文包含稳定挑战标记
- **当** JM 响应正文超过 500 字节且包含 Cloudflare 挑战平台或 “Just a moment” 等稳定标记
- **那么** 系统必须将响应分类为反爬挑战

#### 场景:正常长页面不被误判
- **当** JM 响应为正常 HTML 且不包含挑战响应头或稳定挑战页标记
- **那么** 系统禁止将其分类为反爬挑战

#### 场景:收藏夹内容优先于弱 challenge 文本
- **当** JM 收藏夹 HTML 中存在可解析的收藏夹条目或专辑链接
- **且** 页面仅包含 `captcha` 等弱 challenge 文本
- **那么** 系统禁止将其分类为反爬挑战
