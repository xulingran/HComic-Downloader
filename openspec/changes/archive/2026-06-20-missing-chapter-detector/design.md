## 上下文

工具箱的「重复检测」（`src/components/tools/DuplicateDetector.tsx`）已建立一套成熟范式：拉取收藏夹全量 → `findDuplicateGroups`（LCS≥0.6 + 并查集聚类）→ 按组渲染可折叠卡片。其算法层 `src/utils/titleSimilarity.ts` 是纯函数：`normalizeTitle`、`lcsRatio`、`findDuplicateGroups`、`groupFingerprint`、`extractAlbumTitle`（从一组标题提取共有字段作为专辑名）。

本变更引入「查缺补漏」工具，与重复检测共享算法层，但解决相反问题——重复检测找"多收"（标题相似的若干本），查缺补漏帮用户发现"可能漏收"（同系列聚合 + 一键搜索补全）。

**v1 → v2 方向调整**：v1 尝试用"序号提取 + 缺失推断"自动报告"缺第 N 话"，并为此新增了 `chapterIndex.ts`（正则提取序号）与 `missingChapters.ts`（缺失推断）。真实收藏夹测试暴露了死结：标题命名习惯过于多样（jmcomic 上 `LEVEL:N`、`#N`、`其のN`、`(N)` 并存），即便扩充正则也无法做到可靠覆盖，大量组"无任何序号被提取"。v2 转向务实方案：复用聚类做"同系列聚合展示"，让用户人工判断 + 一键搜索补全。v1 的序号提取/缺失推断代码已删除。

**关键数据约束**：收藏夹分页返回的 `ComicInfo` 在所有来源都不携带结构化 `album_id`/`chapters[]`。要判断"同一系列"，唯一可行手段是标题相似度聚类（与重复检测同源）。

## 目标 / 非目标

**目标：**
- 把收藏夹中"疑似同系列"的条目聚合展示，让用户快速看到哪些可能属于同一作品。
- 为每组提供"搜索此系列"入口，复用既有 pendingSearch 机制一键跳搜索页，方便用户在结果里找漏收的。
- 100% 复用 `titleSimilarity.ts` 算法层（聚类 + 系列名提取），不新增任何算法代码。
- 保持与重复检测同构的交互模式，降低用户学习成本。

**非目标：**
- **不**读 `download_history`，不探测硬盘文件。
- **不**新增任何 IPC 方法或后端改动。纯前端。
- **不**自动报告"缺第 N 话"——v1 尝试过，序号提取覆盖率是死结，已放弃。是否漏收由用户人工判断。
- **不**修改重复检测（`duplicate-detector` capability）的任何行为。
- **不**引入持久化配置（"已确认"标记等）。作为后续增强候选。

## 决策

### 决策 1：v2 方向 = 同系列聚合 + 人工判断 + 搜索补全，放弃自动序号推断

**选择**：复用 `findDuplicateGroups` 聚类展示同系列组，每组提供"搜索此系列"按钮。不尝试自动报告缺失。

**理由**：
- v1 的序号提取（`chapterIndex.ts`）在真实收藏夹上覆盖率不足：jmcomic 标题用 `LEVEL:N`、`#N`、`其のN`、`(N)` 等多种命名，正则清单无法穷尽。即便扩充（v1.1 加了 LEVEL/STAGE/#N/其のN/括号裸数字），仍有大量组"无任何序号被提取"，功能对用户等同失效。
- 自动推断"缺第 N 话"还依赖"站点有权威总章数"这一前提，而收藏夹分页不携带此信息，推断本质是猜测。
- 务实替代：把同系列聚到一起（聚类已成熟可靠）+ 一键搜索补全（复用既有机制），把"判断是否漏收"和"找补漏的"都交给用户 + 搜索引擎。绕开死结，且搜索质量远高于本地正则。

**考虑过的替代方案**：
- *继续扩充序号正则*：覆盖率永远追不上真实命名多样性，维护成本高、收益递减。**否决**。
- *为 jmcomic/bika 利用结构化 album_id 走精确路径*：能解决部分来源，但 hcomic/moeimg 仍无解，体验割裂。**暂缓**。

### 决策 2：组件组织 = 独立组件，fork 骨架而非耦合

**选择**：新建 `MissingChapterDetector.tsx` + `MissingGroup.tsx`，fork `DuplicateDetector` / `DuplicateGroup` 的 UI 骨架（来源切换、分页拉取、进度文案、可折叠组渲染范式），但聚焦"同系列聚合 + 搜索补全"语义。在 `ToolboxPage` 第 4 个入口。共享纯函数原样复用，不重构 `DuplicateDetector`。

**理由**：独立组件让两个功能各自演进不互相干扰。fork 而非复用 React 组件，是因为组的语义不同：重复检测展示"哪几本像 + 相似度%"，查缺补漏展示"同系列成员 + 搜索入口"。

### 决策 3：搜索词来源 = extractAlbumTitle + 清洗

**选择**：用 `extractAlbumTitle(组内所有标题)` 提取系列名，再经 `cleanSeriesNameForSearch` 清洗，作为搜索词。

**理由**：
- `extractAlbumTitle` 是项目已有算法（为"批量下载专辑命名"设计），求组内标题的 token 交集，能可靠提取共有作品名部分。直接复用零成本。
- 但它会把所有成员共有的方括号标记（如 `[中国翻訳]` `[DL版]`）也当交集保留（因为它们确实在所有成员中出现）。这些是版本/翻译噪声，带进搜索词会严重降低搜索质量。例如 LEVEL 系列提取出 `[にのこや (にの子)] エルフに淫紋を付ける本 [中国翻訳] [DL版]`，理想搜索词应是 `エルフに淫紋を付ける本`。
- 故需要 `cleanSeriesNameForSearch` 后处理：剥离所有 `[…]` 方括号标记 + 行尾序号标记（LEVEL:N / 第N话 / #N 等），保留作品名主体。
- 提取失败或清洗后长度 < 2 时禁用搜索按钮（边界保护）。

**清洗策略**：
- 剥离所有方括号标记（反复迭代处理连续/嵌套）。
- 剥离行尾序号标记：英文标签（LEVEL/STAGE/PHASE/PART/ACT/Vol/EP/Chapter）+ 数字、中文（第N话/卷/章/弾）、井号（#N）、日文（其のN）。
- trim 后长度 ≥ 2 才作为搜索词。

**考虑过的替代方案**：
- *直接用组内第一本标题作搜索词*：含太多噪声（作者前缀、版本标记、序号），搜索质量差。**否决**。
- *让用户手输搜索词*：增加操作步骤，违背"一键补全"目标。**否决**。

### 决策 4：搜索触发 = 复用 pendingSearch 机制

**选择**：点击「搜索此系列」调用 `useDrawerStore.setPendingSearch(cleanedName, 'keyword')`，由 `App.tsx` 的 effect 自动跳转搜索页，`SearchPage` 的 effect 自动执行搜索。

**理由**：
- 项目已有成熟的跨页面搜索触发机制，先例是 `ComicInfoDrawer`（点击 tag/作者跳搜索）。完全复用，零新增基础设施。
- `setPendingSearch` 一步完成"跳页 + 搜索"，调用方无需关心路由或搜索执行细节。
- `mode` 用 `'keyword'`（项目无专用"系列"模式；作者用 `'author'`，tag 用 `'tag'`）。

## 风险 / 权衡

- **[相似度聚类的固有误分组]** → 标题像但非同系列的作品被聚到一组。**缓解**：UI 标注"推测"性质；用户可逐组人工核对。与重复检测共享同一局限。
- **[系列名清洗过度或不足]** → 清洗可能误删作品名的一部分，或残留噪声。**缓解**：清洗只剥离明确的方括号标记和行尾序号标记，保留中间所有内容；测试用真实 LEVEL 系列样本验证搜索词为 `エルフに淫紋を付ける本`。
- **[extractAlbumTitle 提取失败]** → 组内成员标题差异大时返回 null。**缓解**：搜索按钮禁用，用户仍可手动展开组查看成员、手动去搜索。
- **[收藏夹大用户的拉取延迟]** → 500+ 本需拉 20+ 页。**缓解**：与重复检测同构，沿用分页进度文案与失败页跳过。

## 迁移计划

- **部署**：纯增量变更，无数据迁移、无配置变更、无后端改动。新增工具箱入口与独立组件。
- **回滚**：移除 `ToolboxPage` 第 4 入口及新增文件即可，零风险。
