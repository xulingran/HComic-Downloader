## 上下文

收藏夹页面当前把来源切换放在标题旁的原生 `<select>` 中，页面主体是占满内容区的卡片网格。来源状态、分页缓存和加载逻辑集中在 `FavouritesPage` 与 `useFavouritesStore`：缓存按 `source + page` 隔离，但当前来源切换代码内联在 `<select onChange>` 中，并使用切换前的 `cache.currentPage` 查找新来源缓存。

应用本身已有一个可展开的全局导航侧栏；设置页又提供了成熟的 150px sticky 内层侧栏样式。收藏夹新增内层侧栏后，内容区在全局侧栏展开时会进一步变窄，现有 `lg:grid-cols-5` 和不换行的顶部工具栏需要一起调整。首次来源选择弹窗、持久化默认来源、分来源缓存与预加载中断均已有规范和测试，本变更应在不改变这些核心语义的前提下替换常规来源切换入口。

## 目标 / 非目标

**目标：**

- 用常驻左侧来源侧栏替换顶部来源下拉框，让用户一眼看到并快速切换四个受支持的收藏来源。
- 来源列表只从 `SOURCES_WITH_FAVOURITES` 派生，避免能力不支持的来源进入 UI。
- 抽取统一的来源切换路径：切换到目标来源第一页，优先显示缓存，并阻止旧来源迟到请求覆盖当前页面。
- 保留默认来源、首次选择弹窗、刷新、分页、批量下载和相邻页预加载的既有行为。
- 在全局侧栏展开与中等窗口宽度下保持卡片和操作区可用。

**非目标：**

- 不修改 `defaultFavouriteSource` 的持久化语义；侧栏点击只改变当前查看来源。
- 不预取所有来源，也不在侧栏展示收藏数量或认证状态。
- 不修改 Python 解析器、下载器、Electron IPC 或配置契约。
- 不给来源侧栏增加收起态、拖拽宽度或独立持久化状态。
- 不移除首次来源选择弹窗。

## 决策

### 决策 1：使用独立展示组件承载来源侧栏，数据与加载仍由页面持有

新增 `FavouriteSourceSidebar` 展示组件，组件内部从 `SOURCES_WITH_FAVOURITES` 与 `SOURCE_LABELS` 派生按钮，只接收 `activeSource: string | null` 和 `onSelect(source)`。侧栏不读取缓存、不发 IPC 请求，也不拥有来源状态。

这样可以让 500 行以上的 `FavouritesPage` 保持清晰，同时避免把业务状态复制进第二个 store。来源按钮使用原生 `<button>`、`aria-current="page"` 和清晰的 focus 样式；当前来源整行使用 accent 背景，未选择状态不高亮任何按钮。

**替代方案：**直接把侧栏 JSX 内联进页面。改动文件更少，但会继续扩大页面组件，并让来源白名单与视觉测试更难隔离，因此不采用。

### 决策 2：页面布局复用设置页的固定宽度 sticky 内栏模式

收藏夹根容器改为横向 `flex`：左侧 `w-[150px] shrink-0`，内部导航 `sticky top-6`；右侧 `flex-1 min-w-0` 承载标题、工具栏、错误状态、卡片与分页。侧栏标题使用“收藏来源”，按钮视觉层级与设置页导航一致。

不为来源制造缺乏依据的品牌图标，首版以短文本标签为主，避免新增图片资源或图标依赖。

**替代方案：**复用应用全局侧栏或把来源做成顶部横向 tabs。全局侧栏承担页面导航，不应混入页面内筛选；横向 tabs 在四项时可行，但与用户要求的设置页式侧栏不符，也会继续挤压顶部操作区。

### 决策 3：统一来源切换到目标来源第一页并复用 `loadFavourites`

从 `<select onChange>` 抽取 `handleSourceChange(newSource)`：

1. 若当前已选择同一来源则不重复请求。
2. 更新本地 `source`、清除 `noSourceSelected`，并调用 `cache.setCurrentSource(newSource)`。
3. 以显式参数调用 `loadFavourites(1, newSource)`。
4. `loadFavourites` 继续负责第一页缓存命中、后台刷新、下载状态检查、登录错误与普通错误展示。

显式使用第一页与 store 的 `setCurrentSource()` 语义一致，也消除当前使用旧来源页码查新来源缓存的不确定行为。来源选择弹窗的 `onSelect` 在标记会话标志后复用同一处理函数。

**替代方案：**为每个来源保存最后浏览页并在切回时恢复。该体验有价值，但需要新增 `lastPageBySource` 状态和更广的分页语义变更，不属于本次快速切换入口改造。

### 决策 4：用请求代次保护用户加载结果，缓存写入与可见 UI 更新分离

快速连续点击来源时，多个第一页请求可能并发返回。现有 `latestPageRef` 只比较页码，无法区分 `hcomic:1` 与 `jm:1`。页面新增递增的请求代次或等价的 `source:page` 上下文令牌：

- 每次用户加载建立新的活动令牌。
- 请求完成后仍可按原始来源写入对应缓存。
- 只有令牌仍为最新且页面仍挂载时，才允许更新当前 `comics`、`pagination`、`needsLogin`、`currentPage` 与下载状态。
- 相邻页预加载继续使用 `usePaginatedPreloader` 的 AbortSignal，不重复实现取消机制。

该保护保证迟到的旧来源结果不会把侧栏高亮与内容区域撕裂成不同来源。

### 决策 5：来源列表与选中状态严格由能力和选择状态派生

侧栏必须严格遍历 `SOURCES_WITH_FAVOURITES`，不再使用返回全部 `COMIC_SOURCES` 的 `useSources()`。顶部 `<select>` 完全移除。

正常状态以 `source` 作为 active 值；用户关闭首次选择弹窗且尚未选择来源时传入 `null`，确保默认初始化值 `hcomic` 不会被误显示为用户已选择。侧栏点击仅更新会话内当前来源，禁止调用设置持久化 API。

### 决策 6：通过更保守的断点和可换行工具栏吸收宽度损失

卡片模式从 `grid-cols-2 sm:grid-cols-3 md:grid-cols-4 lg:grid-cols-5` 调整为 `grid-cols-2 sm:grid-cols-3 md:grid-cols-3 lg:grid-cols-4 xl:grid-cols-5`，避免视口刚进入 `lg` 时在双侧栏夹持下强塞五列。详细列表模式不变。

顶部标题/操作区和内部按钮组增加 `flex-wrap` 与间距；分页仍可位于右侧，但空间不足时必须自然换行，禁止水平溢出或覆盖批量操作。

**替代方案：**使用 `auto-fill + minmax()` 让网格完全按容器宽度响应。它会在超宽窗口生成超过五列，改变现有信息密度；引入容器查询则会扩大 Tailwind 配置范围，因此本次沿用断点模型。

## 风险 / 权衡

- **[风险] 两层侧栏在窄窗口占用较多横向空间** → 来源侧栏保持 150px 固定窄宽，网格减少中等断点列数，工具栏允许换行；不在本次引入第三种折叠状态。
- **[风险] 快速切换造成旧请求覆盖新来源** → 引入用户加载请求代次，只允许最新上下文更新可见 UI，旧结果最多写入自己的缓存。
- **[风险] 移除 `<select>` 后既有测试和键盘操作入口变化** → 使用语义化按钮、可见焦点和 `aria-current`，测试改为按角色/名称点击并验证行为。
- **[权衡] 切换来源回到第一页** → 行为简单、可预测，并与 store 当前重置语义一致；代价是切回来源不会恢复最后浏览页。
- **[风险] 首次弹窗背后可能短暂显示默认 hcomic 高亮** → 未选择状态显式传 `activeSource=null`，弹窗关闭后也保持无高亮，直到用户真正选择。

## 迁移方案

这是无数据迁移的前端替换：先增加侧栏组件和测试，再把页面切换入口接入统一 handler，最后移除 `<select>` 并调整布局。发布后用户配置、缓存结构和后端接口保持兼容。

若需要回滚，可恢复顶部 `<select>` 与原页面根布局，并删除展示组件；不需要回滚配置或数据库数据。
