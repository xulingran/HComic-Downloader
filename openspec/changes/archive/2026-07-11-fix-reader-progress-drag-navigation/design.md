## 上下文

阅读器视觉外壳已经统一在 `ReaderShell`，进度条的指针坐标换算与 pointer capture 也已统一在 `useSliderDrag`。然而“目标页状态如何变成用户可见页面”的编排仍分别位于两个 modal：在线 `ComicReaderModal` 在连续滚动模式下拥有 `scrollToPage`、拖动期冻结和释放后解冻逻辑，本地 `LocalLibraryReaderModal` 只调用 `setCurrentPage`，并且向 `usePageTracking` 固定传入 `false`。因此本地连续滚动阅读时，拖动只改变状态，正文不跳转，可见页观察器还会与拖动写入竞争。

该缺陷不涉及 Python、Electron IPC 或图片来源；修复点完全位于 React 阅读器交互层。约束是保持在线/本地的数据加载策略独立，且不破坏单页、双页模式已有的 `PageFlipView` 翻页与补白页语义。

## 目标 / 非目标

**目标：**

- 让在线预览与本地漫画库阅读器的底部进度条都能按下定位并持续拖动到目标页。
- 连续滚动模式下同步滚动正文，并避免 `IntersectionObserver` 在拖动期间覆盖目标页。
- 单页和双页模式下保持即时状态切页，目标始终限制在有效页码范围内。
- 统一拖动生命周期与清理语义，避免两个阅读器再次发生行为漂移。
- 用用户可见结果而非仅 setter 调用验证回归。

**非目标：**

- 不增加跨章节的一条连续总进度；滑块范围仍限于当前章节。
- 不改变页码到百分比的现有离散映射，也不增加动画式平滑滚动。
- 不改变阅读历史、本地阅读进度持久化、图片缓存或预加载窗口算法。
- 不重新设计进度条外观，也不扩展键盘无障碍操作范围。

## 决策

### 决策 1：抽取共享的阅读器进度导航编排 Hook

在现有 `useSliderDrag` 之上增加共享编排层（建议命名 `useReaderProgressNavigation`），由它组合以下职责：

1. 调用 `useSliderDrag`，把指针位置转换为受限的目标页并更新 `currentPage`。
2. 在 `displayMode === 'scroll'` 且内容已加载时，将拖动中的目标页同步到对应 `pageRefs` 元素。
3. 暴露拖动状态与一个页追踪冻结 ref，供 `usePageTracking` 同时门控。
4. 拖动结束、取消或丢失 pointer capture 后延迟解除冻结，让滚动产生的观察器通知稳定后再恢复正常追踪。
5. 将最终目标页交给调用方既有的 `onDragEnd`，在线版继续触发远程预加载，本地版继续触发附近页物化。

选择共享编排而不是只在本地 modal 复制在线代码，是因为缺陷正是两套编排发生漂移造成的。`ReaderShell` 仍只负责 DOM 与事件绑定，不承载页面状态或滚动策略。

替代方案是仅把在线版第 371–390 行逻辑复制到本地版。改动更小，但会继续保留两套冻结时序和清理代码，后续修改容易再次漏掉一端，因此不采用。

### 决策 2：连续滚动使用即时、按页滚动

拖动产生新的离散页码时，对 `pageRefs.current[targetPage - 1]` 调用即时 `scrollIntoView({ behavior: 'instant', block: 'start' })`；同一页内的多次 pointer move 不重复滚动。即时滚动能让进度条位置与正文反馈保持一致，并避免拖动过程中堆积多段 smooth-scroll 动画。

若目标页 DOM 尚未挂载，状态仍保留目标页，编排层不得抛错；现有渲染完成或后续拖动事件可再次命中。在线与本地滚动模式当前都会渲染全部页面容器，因此正常路径下目标 ref 应存在。

替代方案是仅在 pointerup 时滚动。它减少滚动次数，但失去“拖动时预览目标位置”的快速导航反馈，不符合持续拖动的交互目标。

### 决策 3：拖动状态和显式冻结共同门控页追踪

`usePageTracking` 必须同时尊重共享 Hook 返回的 `isDragging` 和 `freezeRef`。拖动开始后先冻结观察器写入，再更新目标页；拖动结束后保留短暂冻结窗口，防止 `scrollIntoView` 引起的最后一批异步观察通知把页码写回旧值。窗口结束后恢复正常滚动追踪。

冻结只阻止观察器更新 `currentPage`，不停止图片懒加载、预加载或阅读进度保存。单页/双页没有滚动观察竞争，但统一生命周期仍用于滑块视觉状态与最终回调。

替代方案是在每次观察回调中比较事件时间戳或滚动方向。该方案更复杂且容易受浏览器调度影响，显式冻结状态更符合现有在线阅读器已验证的模式。

### 决策 4：测试必须断言导航结果

回归测试分两层：

- Hook/交互层验证 pointerdown、pointermove、pointerup、pointercancel 和 lost capture 的页码计算、持续更新与清理。
- 阅读器集成层使用真实可变状态或状态化测试替身，验证连续滚动模式的目标页面元素实际收到滚动调用，并验证拖动期间触发观察器回调不会把页码改回；单页/双页验证 `PageFlipView` 接收到目标页。

现有仅断言 `setCurrentPage` mock 被调用的测试可保留为坐标换算覆盖，但不能作为快速跳转的唯一证据。

## 风险 / 权衡

- **[高频 pointermove 导致过多渲染和滚动]** → 仅当离散目标页变化时更新与滚动；不在同一页内重复工作。
- **[拖动结束后观察器短暂失真]** → 使用有限延迟解除冻结，并在解除后继续由正常可见页追踪校准。
- **[目标页 ref 暂时不存在]** → 空值时安全跳过滚动且保留目标状态，不让交互崩溃。
- **[抽取共享 Hook 影响在线版既有行为]** → 保留在线版现有即时滚动和约 200ms 解冻语义，并以在线、本地两组集成测试约束等价行为。
- **[pointercancel 与 pointerup 竞态导致重复清理]** → 清理操作必须幂等；最终回调只在正常完成且存在有效目标时执行。

## 迁移计划

1. 先引入共享 Hook 与其单元测试，不改变后端或持久化数据。
2. 将在线阅读器现有进度导航编排迁移到共享 Hook，运行现有阅读器测试确认行为不变。
3. 让本地阅读器接入同一 Hook，并补充连续滚动、单页和双页回归测试。
4. 运行 TypeScript 类型检查、前端测试、ESLint 与测试质量闸门。

回滚只需恢复两个 modal 原有 Hook 组合；没有数据迁移、配置迁移或 IPC 兼容问题。

## 开放问题

无。当前交互范围、模式差异和清理策略均可由现有在线阅读器行为确定。
