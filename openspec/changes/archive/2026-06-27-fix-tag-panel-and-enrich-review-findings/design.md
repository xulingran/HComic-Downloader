## 上下文

NH 标签目录的后端已经支持 `popular` 与 `name` 排序，搜索 Mixin 也能在 `tag` 模式下构造精确标签查询。然而，前端将标签目录与收藏标签合并后固定按 count 降序排序，使 A-Z 结果失效；标签加载 effect 也没有取消或版本校验，较慢的旧请求可以覆盖较新的来源/排序结果。标签弹窗沿用当前搜索 mode 发起搜索，因此从 NH 最近更新或热门排行中选标签时不会进入后端的精确标签分支。

详情抽屉已经具备 `idle/loading/success/error` enrich 状态，但当前把 `loading` 和 `error` 共用同一块红色失败 UI，导致正常请求期间误报失败。

## 目标 / 非目标

**目标：**

- 标签合并后的最终顺序严格服从当前 `popular` 或 `name` 选择。
- 标签列表只允许当前来源与当前排序对应的最新请求修改 UI 状态。
- NH 标签弹窗从任意搜索上下文选中标签后都执行精确标签搜索。
- enrich 的加载提示与失败反馈语义一致，只有真实错误才提供重试。
- 用确定性的回归测试覆盖四个审查问题。

**非目标：**

- 不改变标签数据库、Electron IPC 通道或 NH 远端 API。
- 不重做整个搜索页状态模型，也不引入新的请求管理依赖。
- 不改变非 NH 来源现有的标签搜索语义。

## 决策

### 1. 标签合并函数显式接收排序方式

`mergeTagSources` 接收 `TagListSort`：`popular` 使用合并后的 count 降序并以标签名兜底，`name` 使用标签名升序。这样收藏标签计数合并仍由一个函数完成，同时不会在请求完成后破坏后端所要求的最终展示顺序。

备选方案是完全保留后端返回顺序并把仅存在于收藏中的标签追加到末尾，但这会让合并后的 popular count 与顺序不一致，也无法定义收藏独有标签的 A-Z 位置，因此不采用。

### 2. 使用单调请求版本实现 latest-wins

`useTagPanel` 维护请求版本 ref。每次来源、排序或刷新触发新的标签读取时递增版本，并在 `then/finally` 更新 `tags`、`loading` 或 `refreshing` 前确认版本仍为最新。来源发生变化时，本次读取直接使用默认 `popular`，同时同步排序状态，避免先以旧来源的排序发起一次无效请求。

备选方案是使用 `AbortController`，但当前 IPC Promise 没有端到端取消协议；版本校验可以阻止过期结果污染状态，且无需修改 IPC 契约。

### 3. 在前端把 NH 标签选择归一化为 tag 搜索上下文

当当前来源为 NH 且标签选择发生变化时，搜索页将 mode 与对应 ref 设为 `tag`，清除普通 query，并把全部已选标签通过 tag 参数发起搜索。这样后端现有 `_build_nh_tag_query` 只处理明确的标签输入，不必猜测 keyword/ranking 请求中的 tag 参数究竟是过滤条件还是排序标记。

备选方案是在 Python 中让任何非空 tag 参数优先于 mode，但这会混淆普通关键词与精确标签的组合语义，并可能改变其他调用方行为，因此不采用。

### 4. enrich 状态按语义独立渲染

失败块的显示条件收紧为 `enrichState === 'error'`。`loading` 使用中性“标签加载中…”提示且不渲染重试按钮；重试点击后立即回到 loading，直到成功或再次失败。

## 风险 / 权衡

- [旧 IPC 请求仍会在后台完成] → 使用版本校验隔离结果；本次不扩展取消协议。
- [NH 标签选择会离开热门排行上下文] → 这是精确标签搜索的明确语义；清空标签后按标签模式的空查询恢复 NH 默认结果。
- [不同语言环境的字符串排序可能略有差异] → 沿用项目现有 `localeCompare`，并在测试中使用稳定的 ASCII 标签验证 A-Z 行为。
- [刷新与普通加载共享版本可能互相使结果过期] → 只有最新用户意图可以提交结果，刷新后台操作本身仍可完成并更新数据库。

## 迁移计划

无需数据迁移。按前端 Hook、搜索编排、抽屉 UI、测试的顺序实现；若出现回归，可回退这些前端逻辑而不影响标签数据库和 IPC 数据格式。
