## ADDED Requirements

### 需求:在线阅读器必须在整部漫画末尾提供详情尾页

系统必须仅在在线漫画预览阅读器的完整阅读序列末尾追加一个漫画详情尾页。单本漫画的详情尾页必须位于最后一张图片之后；多章节漫画必须只在最后一章的最后一张图片之后显示一次。到达最后一张图片时禁止强制跳转，用户必须通过继续翻页、滚动或进度导航自然进入详情。

#### 场景:单本漫画末页后进入详情

- **当** 用户在线阅读一部单本漫画并从最后一张图片继续向前导航
- **那么** 阅读器必须进入该漫画的详情尾页
- **并且** 最后一张图片首次进入视口时禁止自动切换到详情

#### 场景:前置章节末尾不显示详情

- **当** 用户阅读一部多章节漫画的非末章并从该章最后一张图片继续向前导航
- **那么** 阅读器必须保持既有跨章导航语义
- **并且** 禁止在该章末尾插入漫画详情页

#### 场景:末章末页后进入详情

- **当** 用户阅读多章节漫画的最后一章并从该章最后一张图片继续向前导航
- **那么** 阅读器必须进入整部漫画唯一的详情尾页
- **并且** 禁止显示进入下一章的提示

#### 场景:从历史记录直达末章

- **当** 用户通过历史记录直接打开多章节漫画的最后一章
- **那么** 阅读器必须根据章节 ID 正确识别当前章节为末章
- **并且** 必须在该章最后一张图片后提供详情尾页

#### 场景:本地漫画库阅读器保持不变

- **当** 用户使用本地漫画库阅读器阅读 CBZ、ZIP 或文件夹资产
- **那么** 阅读序列必须仍仅包含本地漫画页面
- **并且** 禁止自动追加在线漫画详情尾页或触发在线详情请求

### 需求:详情尾页必须覆盖详情抽屉的完整内容与操作

详情尾页必须与漫画详情抽屉共享同一套详情内容、补全状态和操作语义，包括标题、作者、来源、页数、章节数、更新日期、语言、分类、打开原网页、收藏状态与切换、原著、角色、制作组、标签、详情补全、补全失败重试、标签搜索、推荐与屏蔽、确认操作及用户反馈。详情尾页禁止通过复制一套独立业务逻辑造成与抽屉行为漂移，但不需要显示抽屉遮罩、右滑外壳或抽屉关闭栏。

#### 场景:展示完整基础元数据

- **当** 详情尾页接收到含作者、分类、日期、语言、原著、角色、制作组和标签的漫画
- **那么** 尾页必须展示与详情抽屉相同的全部可用字段和来源/页数/章节信息
- **并且** 必须提供打开原网页操作

#### 场景:缺失详情时按需补全

- **当** 当前来源需要详情补全或入口漫画缺少标签，且详情尾页进入或接近可见区域
- **那么** 系统必须通过既有漫画详情 IPC 补全数据并更新尾页
- **并且** 在尾页尚不可见时禁止仅因连续滚动列表已挂载而提前发起补全请求

#### 场景:详情补全失败后重试

- **当** 详情补全抛出异常或返回空漫画
- **那么** 尾页必须显示可见的加载失败状态和重试入口
- **并且** 用户点击重试后必须只重新执行详情补全，不关闭阅读器或重置阅读进度

#### 场景:收藏与标签操作保持一致

- **当** 用户在详情尾页加入或移除收藏，或对标签执行推荐、取消推荐、屏蔽、取消屏蔽
- **那么** 系统必须遵循详情抽屉既有的来源能力门控、服务端结果提交、互斥检查、确认弹窗和成功/失败反馈

#### 场景:详情搜索路由

- **当** 用户点击作者、分类、原著、角色或制作组执行替换式搜索
- **那么** 系统必须写入对应搜索条件并关闭在线阅读器，以显示搜索结果页
- **当** 用户点击普通标签执行追加式搜索
- **那么** 系统必须追加该标签、保留阅读器打开并显示操作反馈

### 需求:详情尾页必须适配三种阅读显示模式

连续滚动、单页和双页模式都必须能够到达并完整操作详情尾页。连续滚动模式必须把详情作为最后一张图片之后的文档块；单页模式必须把详情作为独立尾页；双页模式必须让详情独占整个阅读视口，禁止与漫画图片组成左右半页。

#### 场景:连续滚动到详情

- **当** 用户在连续滚动模式中滚过最后一张漫画图片
- **那么** 详情内容必须作为后续文档块自然进入视口
- **并且** 详情块宽度禁止受漫画图片宽度设置强制限制

#### 场景:单页模式翻入详情

- **当** 用户在单页模式的最后一张漫画图片继续向前翻页
- **那么** 阅读器必须播放既有翻页过渡并显示独立详情页

#### 场景:双页模式详情独占视口

- **当** 用户在双页模式从包含最后一张真实图片的 spread 继续向前翻页
- **那么** 详情必须独占整个内容视口
- **并且** 无论真实图片页数为奇数或偶数、补白位置为 `none`、`front` 或 `end`，详情都禁止与图片或补白页并排

#### 场景:从详情返回最后图片位置

- **当** 用户从详情尾页执行上一页导航
- **那么** 单页模式必须返回最后一张图片
- **并且** 双页模式必须返回包含最后一张图片的最后一个合法 spread

#### 场景:在详情页切换显示模式

- **当** 用户停留在详情尾页并在连续滚动、单页或双页模式之间切换
- **那么** 模式切换完成后必须仍以详情作为可见锚点
- **并且** 禁止被页码钳制逻辑跳回最后一张图片

### 需求:详情尾页交互必须优先于翻页覆盖层

详情尾页中的链接、收藏按钮、标签按钮、确认操作和内部滚动必须完整接收指针及键盘输入。详情激活时，翻页点击热区禁止覆盖或截获详情交互。分页模式下的长详情必须可以在页内纵向滚动，并必须提供可访问的返回最后图片入口。

#### 场景:点击详情页边缘区域内的操作

- **当** 详情页中的收藏、标签或链接操作位于原左右翻页热区覆盖范围内并被用户点击
- **那么** 对应详情操作必须执行
- **并且** 禁止因点击热区而触发上一页或下一页

#### 场景:长详情页内部滚动

- **当** 分页模式下详情内容高度超过可用视口且详情滚动容器仍可沿滚轮方向滚动
- **那么** wheel 输入必须滚动详情内容
- **并且** 禁止触发父级翻页

#### 场景:详情顶部向上滚轮返回

- **当** 详情滚动容器已位于顶部且用户向上滚动
- **那么** 阅读器必须允许该输入返回最后一个图片位置
- **并且** 详情底部继续向下滚动必须保持在详情终点，禁止产生越界页码

### 需求:详情尾页必须与图片副作用隔离

详情尾页必须是前端导航项而非图片页。系统禁止为详情发起预览图片请求、写入图片缓存、加入失败页统计或把详情页位持久化为阅读历史。进入详情后，历史记录必须保持为当前章节最后一张真实图片，历史恢复也必须落在真实图片范围内。

#### 场景:进入详情不请求图片

- **当** 用户从最后一张图片进入详情尾页
- **那么** `fetchPreviewImage`、图片预加载和失败页集合禁止接收详情页索引或哨兵 URL

#### 场景:在详情页停留后关闭阅读器

- **当** 用户进入详情尾页、停留超过历史记录防抖时间后关闭阅读器
- **那么** 写入历史的 `lastPage` 必须等于当前章节最后一张真实图片页码
- **并且** `totalPages` 必须保持真实图片总数

#### 场景:从详情状态恢复历史

- **当** 用户之后通过历史记录重新打开该漫画
- **那么** 阅读器必须恢复到最后一张真实图片或合法的已存图片页
- **并且** 禁止直接恢复到详情尾页或不存在的 `N+1` 图片页
