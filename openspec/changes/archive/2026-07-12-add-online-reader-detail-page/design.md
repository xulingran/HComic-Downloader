## 上下文

在线预览由 `ComicReaderModal` 管理图片 URL、章节选择、显示模式、进度导航、历史记录和预加载；本地漫画库使用独立的 `LocalLibraryReaderModal`。在线翻页分支把 `imageUrls` 与 `totalPages` 直接交给 `PageFlipView`，后者当前假设每一个可导航页位都对应一张图片。连续滚动分支则直接遍历 `imageUrls`，`usePageTracking` 通过页面 DOM ref 更新当前页。

漫画详情由 `ComicInfoDrawer` 独占实现。它既包含抽屉外壳，也包含详情补全、收藏状态、标签推荐/屏蔽、搜索路由、确认弹窗和 Toast 等状态与行为。直接在阅读器复制 JSX 会形成两套容易漂移的详情功能；直接打开现有抽屉又会产生两个同为 `z-50` 的全屏表面、Escape 监听和遮罩竞争，且不符合“详情是阅读序列中的一页”这一交互。

详情尾页不是图片。它必须可被翻页、滚动和进度条定位，但不能进入图片请求、缓存、失败页统计或持久化历史页码。多章节漫画还要求它仅属于整个漫画的结尾，而不是每个章节的结尾。

## 目标 / 非目标

**目标：**

- 仅为在线预览阅读器增加一个可自然到达的详情尾页。
- 单本漫画在末图后显示；多章节漫画只在末章末图后显示一次。
- 详情尾页与详情抽屉共享完整内容、状态和操作语义，不复制业务逻辑。
- 连续滚动、单页和双页模式都能稳定进入、离开和切换到详情尾页。
- 将“真实图片页数”和“可导航阅读项”分开，保护历史、恢复、预加载和错误统计的图片页语义。
- 保持键盘、滚轮、点击热区、滑块和 reduced-motion 的既有门控与可访问性。

**非目标：**

- 不给 `LocalLibraryReaderModal` 增加详情尾页。
- 不修改 Python 解析器、预览 IPC、共享 `ComicInfo` 类型或来源代理行为。
- 不在每章末尾重复显示详情，也不在末图进入视口时强制跳转。
- 不改变详情抽屉已有字段、收藏规则、标签推荐能力门控或搜索语义。
- 不把详情内容渲染为图片、截图或后端生成页面。

## 决策

### 1. 抽取共享详情体验，而不是复用整个抽屉或复制 JSX

把 `ComicInfoDrawer` 中与抽屉几何无关的状态和内容拆为可复用的详情体验层：共享 controller hook 负责详情补全、收藏、标签状态、确认操作和反馈；共享内容组件负责标题、作者、信息、分类、收藏、原著、角色、制作组与标签的同一套 JSX。抽屉仅保留遮罩、右滑面板、标题栏和关闭行为；阅读器尾页提供自己的阅读页容器。

共享详情体验接收以下上下文回调，而不是直接依赖 `closeDrawer`：

- 替换式搜索（作者、分类、原著、角色、制作组）：写入 `pendingSearch` 后由抽屉关闭抽屉、由阅读器关闭阅读器，以便用户看到搜索页。
- 追加式标签搜索：写入组合搜索但保留当前表面，并显示与抽屉一致的反馈。
- `active`：仅在表面实际打开或详情尾页实际进入/接近视口时执行详情补全和收藏状态请求。
- `surface`：`drawer` 与 `reader` 只改变容器尺寸和颜色上下文，不分叉字段及操作逻辑。

确认弹窗和操作反馈属于详情体验的一部分，必须渲染在当前表面的层叠上下文中。这样阅读器尾页不会尝试在阅读器下方打开同层级抽屉。

备选方案：直接在尾页调用 `openDrawer(comic)`。放弃，因为它不是阅读序列中的页面，并会产生层级、Escape 和遮罩冲突。备选方案：复制抽屉 JSX。放弃，因为详情补全与收藏/标签行为后续必然漂移。

### 2. 显式区分图片页数与导航尾页

保留 `useComicReader.totalPages` 为服务端返回的真实图片页数，并在 `ComicReaderModal` 派生：

- `hasDetailTail`：当前是单本，或当前已选章节是章节列表的最后一章；
- `detailPageNumber = totalPages + 1`：仅当 `hasDetailTail` 为真；
- `isDetailPage`：当前导航位置指向详情尾页；
- `navigationTotal`：当前显示模式下包含既有前补白和可选详情尾页的有效终点。

章节索引必须由 `selectedChapterId` 与 `chapters` 派生或在历史直达章节时同步校准，禁止依赖可能仍为 `-1` 的旧状态判断末章。前置章节保持现有“边界再次翻页进入下一章”的行为；末章从最后一个图片 spread 继续前进时先进入详情，不显示“下一章”提示。

任何传给后端、图片缓存、失败页集合或历史存储的页码都使用 `imagePage = min(currentPage, totalPages)`，详情页本身不产生 `fetchPreviewImage` 调用。历史去重也以钳制后的图片页为准，避免停留详情页两秒后写入不存在的 `N+1` 页。

备选方案：向 `imageUrls` 追加哨兵字符串。放弃，因为它会污染预加载、缓存键、失败统计和图片组件类型边界。

### 3. `PageFlipView` 接受尾部内容插槽并对详情使用独占 spread

为在线调用方提供可选的尾部阅读项（内容、语义标签及是否激活），本地调用方不传该插槽。`PageFlipView` 继续使用真实 `imageUrls` 渲染图片，并增加三条特殊导航规则：

1. 当前图片 spread 包含最后一张真实图片且存在详情尾页时，下一次前进跳到详情。
2. 详情尾页始终独占整个视口；双页模式禁止把详情与最后一张图片各塞入半屏，也不为详情再生成补白页。
3. 从详情后退时返回包含最后一张真实图片的合法单页或双页 spread；继续前进保持在详情终点。

详情激活时禁用覆盖内容的左右点击热区，让收藏、标签和链接按钮完整接收指针事件。详情页提供显式“返回末页”入口，同时保留 ArrowLeft/PageUp、滑块和在详情滚动容器顶部向上滚轮的返回路径。详情内容超过视口时使用内部纵向滚动：能继续滚动时消费 wheel；到达顶部且向上滚动时才交给父级执行后退；底部向下滚动不得产生越界翻页。

模式切换协调器把详情视为稳定的语义锚点：用户在详情页切换单页/双页/连续滚动时仍停留详情，而不是被 `clampPage` 拉回最后一张图片。离开详情后再恢复既有图片 anchor 计算。

备选方案：把详情当普通第 `N+1` 页交给 `resolveReaderSpread`。放弃，因为双页奇偶与前补白组合会让详情和图片同屏，部分组合还会被 step=2 越过。

### 4. 连续滚动模式把详情纳入页面追踪，但不纳入图片预加载

连续滚动分支在 `imageUrls.map` 之后追加详情容器，并把它放入 `pageRefs[totalPages]`。`usePageTracking` 的观察目标数量改为导航项数量，使详情进入视口时可把当前导航位置更新为 `detailPageNumber`；`useReaderProgressNavigation` 因此能将滑块末端滚动到详情容器。

图片预加载仍只接收 `imageUrls`，且 `preloadTarget` 在详情页时钳制为 `totalPages` 或跳过更新。详情内容的 `active` 在连续滚动模式下由自身可见性触发，以免仅打开一本长漫画就提前执行详情补全或收藏检查。

### 5. `ReaderShell` 支持语义化页码显示

`ReaderShell` 增加可选的当前项标签/ARIA 文本。当详情激活时，页码区域显示“详情”，进度按终点 100% 呈现，滑块 `aria-valuetext` 表达“漫画详情”；图片页继续显示现有 `current / effectiveTotal`。本地阅读器不传语义标签，渲染和范围完全不变。

滑块最右端必须可定位详情；从详情向左拖动则回到对应图片页位。详情页不伪装成“第 N+1 张图片”。

### 6. 详情尾页使用阅读器暗色表面但保持共享结构

详情内容组件使用语义化 surface token 或由阅读器包装层覆盖局部 CSS 变量。阅读器变体采用暗色背景、受限正文宽度和可滚动高度；抽屉继续使用应用主题变量。禁止通过复制整套 Tailwind markup 制造两种主题。

连续滚动模式的详情块拥有独立最大宽度，不跟随漫画 `imageWidth`；翻页模式占满可用阅读视口但正文保持适合阅读的最大宽度。reduced-motion 继续使用现有阅读器模式/翻页降级，不新增强制动画。

## 风险 / 权衡

- **[双页奇偶、前补白与详情尾页组合复杂]** → 把“最后图片 spread”“详情 spread”和前后导航抽成纯函数，并覆盖奇数/偶数页数及 `none/front/end` 三种补白测试。
- **[共享详情重构可能造成抽屉行为回归]** → 先用现有 `ComicInfoDrawer` 测试固化字段、收藏、标签、搜索和失败反馈，再让抽屉与尾页消费同一共享体验；保留既有抽屉回归用例。
- **[详情页按钮被翻页覆盖层截获]** → 详情激活时关闭边缘点击覆盖层，并为返回末页提供独立可访问入口。
- **[长详情内部滚动与滚轮翻页冲突]** → 根据详情滚动容器的 `scrollTop/scrollHeight/clientHeight` 决定消费或放行 wheel，并测试顶部、正文中部和底部三种边界。
- **[尾页污染历史或预加载]** → 后端与缓存始终只接收真实图片索引，历史写入统一经过页码钳制；加入“详情停留后关闭并恢复”的回归测试。
- **[连续滚动一打开就请求详情]** → 详情 controller 由可见性 `active` 门控，不以组件挂载等同于可见。
- **[历史直达末章时末章判定错误]** → 由章节 ID 反查索引，不把 `currentChapterIndex === -1` 当作非末章的最终依据。

## 迁移计划

1. 先抽取共享详情体验并让现有抽屉改用它，运行抽屉定向测试确认无行为变化。
2. 增加尾部阅读项与纯导航辅助，接入单页/双页并覆盖边界测试。
3. 接入连续滚动、语义化进度和模式切换锚点。
4. 最后接入历史/预加载隔离与多章节末章判定，运行在线及本地阅读器完整回归。

本变更没有持久化数据迁移，可随前端代码整体回滚；回滚尾部插槽后，既有图片页、历史数据和 IPC 契约仍可直接使用。

## 开放问题

无。提案采用已确认语义：详情只在整部漫画末尾出现一次，且通过自然继续阅读进入，不强制跳转。
