## 上下文

JM 搜索页目前存在两条不一致路径：首次挂载会执行空关键词 `search`，而用户切换到已认证 JM 时会直接执行 `random`。`JmParser.search("")` 又会请求空查询搜索 URL，因此没有任何路径能表达 JM 首页的栏目结构。随机漫画也因此从一个显式用户动作变成了切源副作用。

2026-07-03 保存的 JM 首页快照包含 9 个可见栏目。现有 `_parse_search_item()` 可直接解析其中 7 个栏目的标准专辑卡片，整体可得到 86 张卡、80 个唯一漫画；“禁漫书库”和“禁漫小说”没有标准 `/album/{id}` 卡片，不兼容当前 `ComicInfo` 和下载流程。此变更跨越 parser、分发/IPC、共享类型、搜索缓存和 React 渲染，需要显式约定分组模型与状态清理边界。

## 目标 / 非目标

**目标：**

- 让 JM 空白关键词第一页具有稳定的“首页栏目”语义。
- 默认展示五类高价值栏目，每栏最多 10 本，并保留站点动态标题和顺序。
- 复用现有卡片解析、认证 Session、系统代理与挑战恢复机制。
- 用唯一漫画集合承载过滤、选择和下载，用轻量 ID 引用承载栏目分组。
- 保证普通搜索、随机、排行、缓存刷新和其他来源行为不回归。

**非目标：**

- 不下载或解析禁漫小说、禁漫书库等非专辑内容。
- 不实现首页“看更多”跳转、横向轮播或新的栏目分页。
- 不移除 JM 随机功能，也不改变随机按钮的 `/albums/random` 行为。
- 不新增 Electron IPC 通道，不持久化首页栏目，不把首页 HTML 快照返回 renderer。
- 不修改 JM Cookie 生命周期、域名发现或系统代理策略。

## 决策

### 1. 使用 JM 专用首页方法，而不是改变通用 `search()` 返回元组

为 `JmParser` 增加专用首页解析入口，并由 `MultiSourceParser` 暴露有类型的 JM 首页方法。`SearchMixin.handle_search` 仅在 `jm + keyword + trim 后空串 + page=1` 时进入该入口，其他查询继续走现有 `parser.search()`。

选择专用入口是因为通用 `search()` 当前稳定返回 `(list[ComicInfo], PaginationInfo | None)`，强行加入第三个返回值会波及所有来源和大量调用方；让 `search("")` 返回带隐式分组的特殊对象也会破坏 parser 接口一致性。专用入口将特殊性限制在 JM 首页，同时仍复用既有搜索 IPC。

### 2. 通过栏目“看更多”目标识别语义，显示标题取 DOM 文本

解析器从 `h4.talk-title` 找到其最近标题行，再读取紧邻的内容行。默认五类栏目用相对稳定的“看更多”目标形态识别：`/serialization/`、禁漫汉化组搜索、`/albums/hanman`、首个推荐 `/promotes/`、`/albums?o=mr`。返回标题使用页面 `h4` 的规范化可见文本，而不是硬编码“周五”或“C107”。

每栏调用现有 `_parse_search_item()`，保留前 10 个有效且栏内不重复的漫画。栏目没有有效专辑时跳过；单卡解析异常继续沿用现有容错策略。这样既能排除书库/小说，又能适应星期、活动届次与繁简标题变化。

替代方案是按完整中文标题匹配，结构简单但会被星期和活动届次变化击穿；另一个方案是返回所有含专辑卡的栏目，会使默认页过长并把站点临时推广栏目无界带入应用，因此不采用。

### 3. `SearchResult` 使用规范化分组而不是复制漫画对象

共享契约新增：

```ts
interface SearchSection {
  title: string
  comicIds: string[]
}

interface SearchResult {
  comics: ComicInfo[]
  pagination: PaginationInfo
  sections?: SearchSection[]
}
```

Python 先按 `(source_site, id, comic_source)` 去重顶层漫画并保持首次出现顺序，再为栏目序列化有序 ID。JM 单一来源中 ID 可作为引用键；IPC 构造阶段必须验证引用存在并去掉栏内重复 ID。首页分页固定为 1/1。

复制完整 `ComicInfo` 到每个栏目虽然渲染直观，但会重复 JSON 数据，并让批量选择、过滤和下载需要额外去重。给 `ComicInfo` 增加 `section` 字段则会污染跨来源领域模型且无法自然表达同一本漫画属于多个栏目，因此采用规范化结构。

### 4. SearchPage 持有栏目视图状态，缓存与提交路径必须原子更新

`SearchPageCache` 增加可选 `sections`。SearchPage 用局部栏目状态配合顶层 `comics` store：结果提交时同时设置漫画、分页和栏目；缓存恢复时一起恢复；新查询清空列表时同步清空栏目。建议抽取单一 `applySearchResult`/`clearSearchResult` 编排 helper，覆盖挂载自动加载、`withLoading`、缓存后台刷新和 pending search，避免某条分支遗留旧栏目。

渲染时从经过现有黑名单和推荐计算的 `filteredComics` 建立 ID 映射，再按 `sections[].comicIds` 组装每栏卡片。选择状态继续以现有 `getComicKey(comic)` 为准，因此跨栏重复卡片共享选中状态，而 `useBatchDownload(comics)` 只看到唯一顶层漫画。普通结果在 `sections` 缺失或为空时继续进入现有平铺网格。

### 5. 用户主动切源可交互，自动首页加载保持静默

已认证用户主动切换到 JM 时改为调用 `search('', 'keyword', 1, 'jm', undefined, true)`；显式点击空白搜索已经属于用户动作，同样允许交互恢复。首次挂载按默认来源自动加载、缓存刷新与预加载继续缺省 `false`，防止应用启动或后台活动突然弹出验证窗口。

首页请求使用 `_request_text_with_challenge_check()` 请求根 URL。挑战错误携带根 URL，并复用现有 Electron 搜索挑战窗口和一次性重试；不增加首页快照兜底。随机按钮继续调用 `random('jm')`，其既有挑战行为不在本变更范围内。

### 6. 首页栏目仍遵守现有动画与加载反馈

新请求开始时同时清空漫画和栏目，继续显示约 12 张骨架卡。栏目返回后，每个栏目拥有独立网格容器；动画 key 包含搜索上下文、栏目索引/标题和漫画键，避免不同栏目中的相同漫画产生 React key 冲突。每栏的 stagger 索引从零开始且仍受前 20 项限制，reduced-motion 继续由现有组件处理。

不为首页增加分页器或预加载；固定 1/1 分页自然阻止 `useSearchPreloader` 发出相邻页请求。

## 风险 / 权衡

- **[JM 首页 DOM 或“看更多”路径变化]** → 解析器以最小结构关系和路径形态匹配，单栏失败时跳过；使用脱敏的最小首页 fixture 锁定当前契约，避免把 7.7MB SingleFile 快照提交进测试。
- **[栏目结果状态在多条异步提交路径中残留]** → 用统一结果提交/清理 helper，并测试普通搜索、随机、切源、缓存恢复和迟到后台刷新。
- **[同一漫画跨栏目重复渲染导致重复下载]** → IPC 顶层去重、栏目只引用 ID，批量操作只消费顶层唯一数组。
- **[首页 50 张卡增加首屏渲染成本]** → 固定五栏、每栏最多 10 张；继续使用现有图片懒加载、动画上限和卡片组件，不做详情补全风暴。
- **[首页挑战在启动时打扰用户]** → 只有用户主动动作传 `allowInteractiveChallenge=true`，自动和后台调用保持非交互。
- **[栏目识别过于站点特化]** → 特化封装在 `sources/jm` 内，不扩散到通用 parser；普通搜索契约仅增加可选字段。

## 迁移计划

1. 先扩展可选共享契约和缓存类型，旧结果不提供 `sections` 时行为不变。
2. 增加 JM 首页 fixture、解析和 IPC 序列化测试，再接入空白查询分支。
3. 增加 SearchPage 分组渲染与统一状态提交，最后把 JM 切源行为从 random 改为空白 search。
4. 运行 JM/搜索定向测试后执行项目完整七项验证流程。

回滚时可先恢复 JM 切源 random 调用，再删除空白查询首页分支和可选 `sections`；因为没有持久化迁移和新 IPC 通道，回滚不需要数据清理。

## 待确认问题

无。默认栏目集合、每栏上限、交互挑战边界和随机按钮保留策略已在本设计中确定。
