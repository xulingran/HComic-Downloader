## 上下文

主导航使用 `visitedPages` 保存已访问页面，并由每个 `KeepAlivePage` 持有独立的 `AnimationControls`。Tab 切换时，离开页执行 300ms 的 `x: 0 → ±8%`、`opacity: 1 → 0`，已访问的进入页同时从上次退出状态执行相反动画；首次访问页则以 `opacity: 1` 直接挂载。由于两个 `absolute inset-0` 的全尺寸页面在同一层叠上下文中同时可见，而且仅相对移动 8%，绝大部分内容会进行透明度混合，形成用户看到的文字和卡片重影。

现有方案还有一个生命周期风险：退出动画的 Promise 无条件在完成后执行 `setDisplay('none')`。快速切回同一页面时，旧 Promise 可能晚于新的进入命令完成并隐藏已重新激活的页面。现有测试只断言动画目标函数被调用，没有覆盖动画中间态的可见页面数量和过期完成回调。

约束包括：已访问页面必须保持挂载，页面内部状态与滚动位置不能丢失；首屏不能因动画 controls 绑定时序出现白屏；程序化导航与侧边栏点击必须共用行为；overlay 不能进入 Tab 动画容器；不新增依赖。

## 目标 / 非目标

**目标：**

- 在任意 Tab 过渡时刻最多只显示一个页面的真实内容，彻底消除交叉透明导致的重影。
- 保留方向感知的 slide + fade、keep-alive、程序化跳转和 300ms 总过渡节奏。
- 首屏直接可见；首次访问和切回已访问页面使用一致、可控的进入阶段。
- 快速连续点击采用 latest-intent-wins，过期动画完成结果不得改变当前可见页面。
- 建立能够验证中间动画阶段而不只是函数调用次数的回归测试。

**非目标：**

- 不修改页面自身的列表、Drawer、Modal、阅读器或侧边栏动画。
- 不改变 `TAB_ORDER`、路由结构、页面数据加载或 keep-alive 的实例保留语义。
- 不迁移到 React Router、CSS View Transitions API 或新增动画依赖。
- 不处理当前独立变更 `fix-reader-wheel-flip-lock` 的阅读器翻页逻辑。

## 决策

### 决策 1：使用顺序式 fade-through，替代同步 crossfade

一次普通 Tab 切换仍占用 300ms，但拆成两个各 150ms 的阶段：

```text
0ms                         150ms                         300ms
旧页 visible ── exit ──▶ hidden
                              新页 enter-start ── enter ──▶ visible
```

退出阶段只有旧页为 `display:block`；旧页完全淡出后立即设为 `display:none`，再将目标页设为 `display:block` 并播放进入动画。两个页面的 DOM 可以因 keep-alive 同时存在，但它们的真实内容禁止同时可见。

方向仍由 `TAB_ORDER` 索引差决定。向右导航时旧页向左移动 8%，新页从右侧 8% 进入；向左导航时相反。每个阶段使用无 overshoot 的 tween 曲线。总时长仍为 300ms，因此不会把旧 `mode="wait"` 方案的 300ms 退出和 300ms 进入串成 600ms。

考虑过的替代方案：

- 保留同步 crossfade，仅给页面增加背景：背景也会随父容器 opacity 混合，不能从语义上保证内容不重叠。
- 改成 100% 全页推送：可以避免空间重叠，但运动幅度过大，与现有克制的 8% 动画语言不一致。
- 直接取消 Tab 动画：能消除问题，但不满足方向感和切换反馈需求。

### 决策 2：在父级集中协调过渡，页面容器只执行阶段命令

并行动画的根源是各 `KeepAlivePage` 根据自己的 `isActive` 独立启动动画。改为由 `App` 层的 Tab 过渡协调器维护：

- 用户最新请求的 `desiredPage`；
- 当前唯一可见的 `visiblePage`；
- `idle | exiting | entering` 过渡阶段；
- 当前导航方向；
- 单调递增的过渡身份或等价的取消标识。

`KeepAlivePage` 继续持有与自身 motion 元素绑定的 controls，但不再自行推断 false→true / true→false 并同时启动动画。父级只向当前阶段对应的单个页面下发 `exit`、`hide`、`show-at-enter-start`、`enter` 命令。这样可见性不再由多个互不知情的 Promise 分别收尾。

考虑过让所有页面共享一个 `AnimationControls`，但一个 controls 会广播给全部订阅元素，无法独立控制显示状态，因此仍保留每页独立 controls。

### 决策 3：快速导航采用串行协调和 latest-intent-wins

过渡协调器始终记录最新目标：

- 在退出阶段收到新目标时，当前退出继续完成，但进入阶段直接选择最新目标，跳过已经过时的中间页面。
- 在进入阶段收到新目标时，当前进入完成后立即从该唯一可见页面发起下一次过渡；不允许第二个页面同时进入。
- 每个异步完成回调必须核对过渡身份和组件挂载状态；过期回调只能结束自身，禁止修改 `visiblePage`、phase 或任一页面的 display。
- 同页请求不创建新过渡。

该策略最多等待当前 150ms 半阶段，换取确定的单可见页不变量，并避免快速点击时出现多页堆积或当前页被旧 Promise 隐藏。

### 决策 4：首次访问先隐藏挂载，再由协调器进入

应用首屏仍以可见态直接渲染，完全绕过 controls，保留现有白屏防护。首次访问其他 Tab 时，新 `KeepAlivePage` 先以 `display:none` 挂载；旧页退出的 150ms 为 motion 元素完成 controls 绑定提供了稳定窗口。切换到进入阶段后，协调器先同步设置新页的进入起点（`x: ±8%`、`opacity: 0`），再显示并启动进入动画。

这消除了“首次访问页面直接盖在退出页上”的现象，也不再依赖 mount 后立即 `controls.start()` 的竞态窗口。

### 决策 5：reduced-motion 使用瞬时单页切换

检测到 reduced-motion 时不播放 slide 或 crossfade：旧页立即隐藏，目标页立即显示为最终状态。keep-alive 和 latest-intent-wins 仍然生效。瞬时切换既符合减少运动的用户意图，也天然满足不重叠约束。

## 风险 / 权衡

- **[风险] 顺序动画中点出现短暂空白帧** → 在同一次渲染提交中完成旧页隐藏、新页设置进入起点和显示；主内容容器保留统一背景色，避免视觉闪白。
- **[风险] 首次访问页面在隐藏状态挂载仍可能执行数据请求或布局工作** → 维持现有懒创建时机，不额外预挂载；只改变可见性，且既有 idle chunk 预热继续生效。
- **[风险] 快速切换的串行阶段可能让侧边栏选中项短暂领先于页面内容** → 侧边栏表达最新用户意图，内容在最多一个 150ms 半阶段后跟进；测试明确覆盖最终一致性。
- **[风险] 多个 state 更新造成中点闪烁** → 把阶段切换封装为单一协调动作，必要时使用 reducer，禁止让每个页面独立 setState 收尾。
- **[权衡] `App` 增加一个小型过渡状态机** → 换取可证明的单可见页不变量、可取消性和更直接的行为测试。

## 迁移计划

1. 先补充动画目标和过渡协调测试，固定“最多一个页面可见”的失败用例。
2. 引入集中式过渡阶段与 latest-intent-wins，调整 `KeepAlivePage` 的命令接口和 display 所有权。
3. 将动画时长拆为两个 150ms 半阶段，并切换 reduced-motion 行为。
4. 运行 TypeScript、Vitest、ESLint 及 OpenSpec 严格验证。

纯前端内部行为变更，无数据或配置迁移。若需回滚，可整体回退本变更，恢复独立 controls 的同步 crossfade；keep-alive 页面数据不受影响。

## 待决问题

无。fade-through、300ms 总时长、8% 方向位移和 reduced-motion 瞬时切换均作为本变更的确定决策。
