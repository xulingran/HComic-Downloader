## 上下文

登录弹窗浮标叠层（`electron/login-preload.ts`）在成功后进入倒数态，从 `COUNTDOWN_START` 秒倒数到 0 再通过 `LOGIN_FINISH` 请求主进程关窗。该起始秒数由模块级常量 `COUNTDOWN_START` 定义，被 `renderCounting` 函数读取。倒数态同时服务两个场景：

- **普通登录模式**（`mode: 'login'`）：cookie 提取成功 → 倒数关窗。
- **JM 人机验证模式**（`mode: 'challenge'`）：人机验证成功 → 倒数关窗。

两者共用同一个 `renderCounting` / `COUNTDOWN_START`，因此改一处即同时生效。用户反馈 5 秒偏长，缩短到 3 秒。

主进程侧有兜底常量 `LOGIN_FINISH_FALLBACK_MS = 10_000`，用于渲染端因任何原因未在合理时间内发出 `LOGIN_FINISH` 时由主进程收尾关窗。

## 目标 / 非目标

**目标：**

- 将成功后的自动关窗倒数从 5 秒缩短为 3 秒。
- 普通 cookie 提取成功与 JM 人机验证成功两个场景一致生效。
- 同步更新注释、规范文字、单测断言，保持代码与文档一致。

**非目标：**

- 不改变倒数机制本身（仍每秒减 1、仍可取消、倒数到 0 仍调 `LOGIN_FINISH`）。
- 不调整主进程 `LOGIN_FINISH_FALLBACK_MS`（10s 兜底余量仍充足，缩短它无收益且增加误触风险）。
- 不区分两个场景的倒数时长（共用常量是有意设计，二者语义一致）。
- 不涉及 IPC 通道、Python 后端或登录窗口主进程逻辑的任何改动。

## 决策

**决策 1：仅改 `COUNTDOWN_START` 单一常量，两个场景共用。**

考虑的替代方案：
- *按 `mode` 分别设两个常量（如 `LOGIN_COUNTDOWN_START` / `CHALLENGE_COUNTDOWN_START`）*：当前两场景的成功语义完全一致（都是"成功后给用户一个可取消的缓冲再关窗"），没有差异化需求。拆分只会增加无谓的分支与维护成本，因此不采用。
- *用户可配置*：这是一个无关紧要的微调（5→3），不值得引入配置面、持久化与设置 UI 的复杂度。不采用。

理由：保持单常量、最小改动面，与现有架构（两场景共用 `renderCounting`）一致。

**决策 2：不联动调整 `LOGIN_FINISH_FALLBACK_MS`。**

该兜底定时器（10s）用于渲染端崩溃 / 导航丢状态时主进程收尾。即使渲染倒数从 5s 缩短为 3s，10s 兜底仍远超正常倒数路径所需，缩短它只会压缩容错余量而无任何正面收益。维持不动。

## 风险 / 权衡

- **[风险] 倒数过快，用户来不及点「取消」** → 3 秒仍保留完整的中途取消能力（「取消」按钮始终可见），且用户本来就是主动触发提取/验证成功，缩短等待是正向体验。可接受。
- **[风险] 测试断言与代码脱节** → 必须同步更新 `tests/unit/preload/login-preload.test.ts` 中断言数字 `5`（`.count-num` 文本）与时间 `5_000`ms（fake timer 推进）的用例，否则测试会失败。
- **[风险] 文档与实现脱节** → 同步更新 `login-overlay` 规范中「倒数从 5 开始」的两处文字，以及 `login-window.ts` 注释中「渲染端倒数默认 5s」的描述。
