## ADDED Requirements

### 需求:封面异步加载
系统必须使用固定大小（4 个工作线程）的线程池异步加载封面图片，避免阻塞 UI 主线程。

#### 场景:调度封面加载
- **当** 搜索结果卡片需要加载封面
- **那么** 如果缓存命中，直接显示缓存图片；否则提交到线程池，标记 URL 为加载中，防止重复提交

#### 场景:加载完成显示
- **当** 封面图片加载成功
- **那么** 将图片缓存，通过主线程回调更新 Label 显示

#### 场景:加载失败显示重试图标
- **当** 封面图片加载失败
- **那么** 在 Label 上显示重试图标，点击后重新调度加载

#### 场景:过期任务丢弃
- **当** 搜索结果已更新（generation counter 变化）
- **那么** 后台线程中检测到 generation 不匹配时必须立即终止处理，不更新 UI

### 需求:滚动期间延迟图片刷新
系统必须在滚动激活期间缓存图片更新请求，滚动空闲后批量刷新，避免回调风暴。

#### 场景:滚动中收到图片更新
- **当** 封面加载完成但当前处于滚动状态
- **那么** 将 (label, photo) 缓存到待更新队列，每个 label 仅保留最新值

#### 场景:滚动空闲后批量刷新
- **当** 滚动空闲回调触发
- **那么** 遍历待更新队列，在主线程批量更新所有 label 的图片显示

### 需求:封面缓存管理
系统必须根据 URL + 尺寸生成缓存键，管理图片缓存和加载状态。

#### 场景:结果切换时清理
- **当** 新的搜索结果加载或来源切换
- **那么** 清空图片缓存、取消所有待刷新请求、清理加载中标记集合
