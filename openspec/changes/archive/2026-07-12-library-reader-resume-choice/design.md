## 上下文

本地漫画详情由 `LibraryAssetDetailDrawer` 提供阅读入口，`LibraryCatalogView` 获取最新资产详情后把它交给提升到 App 根的 `useLocalReaderStore`。`LocalLibraryReaderModal` 当前总是把详情中的 `readingChapterId` 和 `readingPage` 传给 `useLocalLibraryReader.fetchAsset`，因此已有进度会自动恢复。Python 侧数据库、详情 IPC、页清单校验和节流保存已经完整，本变更只需让渲染进程显式携带本次会话的启动意图。

多章节资产还存在一个重要约束：没有有效保存章节时，当前阅读器会显示章节选择器。选择“从头开始”则必须显式选择当前第一章，不能仅把保存进度清空，否则会意外进入章节选择器。

## 目标 / 非目标

**目标：**

- 在已有有效进度时提供清晰的“继续阅读”和“从头开始”选择。
- 把启动意图从详情入口可靠传递到阅读器初始化，覆盖单本和多章节资产。
- 保持加载失败前的旧进度不变，并复用现有保存、边界限制和退出 flush 机制。
- 对失效章节进度进行安全回退，不构造无效页面请求。

**非目标：**

- 不新增或迁移阅读进度数据库字段。
- 不新增 Python IPC，也不改变本地图片安全交付协议。
- 不改变在线预览阅读器、显示模式、翻页和预加载行为。
- 不增加“每次询问/默认行为”设置或跨设备进度同步。

## 决策

### 1. 使用显式会话启动模式

本地阅读器 store 保存资产详情以及 `resume | restart` 启动模式。详情抽屉通过带模式参数的回调发起打开；阅读器初始化只读取该会话快照，不从按钮文案或进度是否为空反推意图。

选择显式模式而不是在“从头开始”时复制并篡改 `LibraryAssetDetail`，因为后者会让展示数据与后端事实不一致，也难以区分“确实没有进度”和“本次忽略进度”。模式属于 UI 会话状态，不需要加入共享 IPC 契约。

### 2. 从头开始只覆盖初始化参数

`resume` 继续传入保存的章节和页码；`restart` 对单本传入第 1 页，对多章节从详情中的有序章节列表选择第一章并传入第 1 页。旧数据库记录不在点击按钮时删除。

不采用先调用删除进度 IPC 的方案，因为页面清单加载失败、资产变化或用户立即关闭时会不可逆丢失原位置。阅读器成功形成有效页面状态后，现有进度 effect 会自然写入新位置。

### 3. 入口按进度有效性呈现

无 `readingPage` 时显示单一“开始阅读”。存在页码且保存章节对当前章节列表有效（单本允许空章节 ID）时显示两个并列操作，并让“继续阅读”保持主操作视觉层级。“从头开始”使用次级样式，避免确认弹窗增加一次额外交互。

保存章节失效时不把“继续阅读”描述成可用恢复路径：入口允许从头开始；若通过旧快照或竞态进入恢复流程，沿用当前章节选择器回退。相比每次点击弹出二选一对话框，详情内双入口可直接表达当前进度且键盘操作更简单。

### 4. 会话生命周期与现有退出动画一致

启动模式与资产绑定在同一个 store 会话中，直到共享阅读器外壳完成退出动画后再清理。过期退出回调不得清除新会话的启动模式，沿用现有资产会话标识/关闭保护策略。

## 风险 / 权衡

- [详情中的章节或进度在点击后变旧] → 打开前继续重新获取最新详情，并由 `fetchAsset`/页清单接口执行最终章节和页码校验。
- [从头打开后初始化 effect 过早保存第 1 页] → 仅在页面清单已加载且总页数有效的既有条件下保存；测试覆盖清单加载失败时旧进度不被覆盖。
- [多章节第一章排序不稳定] → 使用后端详情已按章节索引返回的 `chapters[0]`，不在前端按名称重新排序。
- [两个按钮增加详情抽屉空间] → 仅在存在可恢复进度时并排显示，窄宽度下允许保持等宽布局和清晰焦点顺序。

## 迁移计划

无需数据迁移。先扩展本地阅读器会话状态和初始化参数，再切换详情抽屉入口，最后运行定向前端测试及完整 TypeScript、Vitest 和 ESLint 验证。回滚前端改动即可恢复当前自动继续阅读行为，已有进度数据不受影响。

## 待解决问题

无。默认采用详情抽屉内双入口，并以“继续阅读”为主操作。
