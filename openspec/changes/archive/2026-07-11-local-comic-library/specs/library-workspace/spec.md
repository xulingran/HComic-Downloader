## ADDED Requirements

### 需求:漫画库工作区包含漫画库与下载任务

系统必须将原“下载管理”入口呈现为“漫画库”工作区，并在工作区内提供“漫画库”和“下载任务”两个子页签。“漫画库”必须是首次进入时的默认子页签；切换子页签禁止卸载或重置仍在运行的下载任务状态。

#### 场景:首次进入工作区
- **当** 用户在本次应用会话中首次点击侧边栏“漫画库”
- **那么** 系统显示“漫画库”子页签及本地资产列表，并同时显示“下载任务”入口

#### 场景:切换到下载任务
- **当** 用户从“漫画库”切换到“下载任务”
- **那么** 系统显示现有下载任务列表，漫画库筛选条件和滚动位置在会话内保持不变

### 需求:保留现有下载管理行为

“下载任务”子页签必须保留既有单任务暂停、恢复、重试、取消、失败详情、全局暂停、状态筛选、专辑折叠、专辑批量控制和强制打包行为；本变更禁止改变既有下载状态集合及其语义。

#### 场景:操作现有下载任务
- **当** 用户在“下载任务”子页签暂停、恢复、重试或取消任务
- **那么** 系统通过既有下载 IPC 执行操作，并按既有状态机更新任务展示

#### 场景:专辑任务保持原有交互
- **当** 下载任务包含多章节专辑
- **那么** 专辑折叠、失败章节置顶和专辑级控制继续满足 `download-album-collapse` 规范

### 需求:跨子页签显示活跃下载提示

工作区必须在“下载任务”页签上显示活跃任务数量；活跃任务判定必须复用 `ACTIVE_DOWNLOAD_STATUSES`，禁止另行枚举状态字面量。

#### 场景:浏览漫画库时下载仍在进行
- **当** 用户停留在“漫画库”且有 3 个处于活跃状态的下载任务
- **那么** “下载任务”页签显示数量 3，任务进度事件继续更新全局下载 store

#### 场景:活跃任务全部结束
- **当** 最后一个活跃任务进入终态
- **那么** 活跃数量立即更新为 0，漫画库视图不被强制切换

### 需求:工作区遵循页面懒创建与保活约束

漫画库工作区必须继续作为 `downloads` 页面参与既有页面懒创建与保活机制；未访问工作区时禁止启动漫画库扫描或加载其页面 chunk，访问后切换到其他侧边栏页面不得重复创建实例。

#### 场景:应用启动但未访问漫画库
- **当** 应用启动后用户尚未访问侧边栏“漫画库”
- **那么** 漫画库页面未挂载，后端未因该页面发起目录扫描

#### 场景:离开后返回工作区
- **当** 用户访问漫画库、切换到其他页面后再返回
- **那么** 系统复用已创建的工作区实例，并执行轻量索引刷新而非无条件完整重扫
