## 上下文

当前下载页读取 `ComicDownloadManager` 进程内任务，后端重启后并不能代表磁盘上的漫画集合。另一方面，`python/maintenance/scanner.py` 已能识别下载目录顶层的 CBZ、ZIP 和图片文件夹，`download_history.db` 保存下载来源及输出路径，维护中心也已有压缩包检查和存储统计逻辑。现有阅读器只消费远程预览 URL；`app-image://` 只从受控封面/预览缓存目录流式交付文件。

该变更横跨 React 页面、Zustand 状态、共享 IPC 契约、Electron 主进程、Python 扫描/持久化和本地文件操作。必须继续满足以下约束：

- Electron 渲染进程处于 Context Isolation + Sandbox 环境，禁止获得任意本地文件读取或删除能力。
- Python 与 Electron 仍使用 JSON-RPC over stdin/stdout；图片字节禁止经过 JSON-RPC。
- 当前 `downloadDir` 是唯一漫画库根目录；目录迁移完成前仍以旧目录为准。
- 本地漫画访问不产生网络请求；现有网络代理要求不受影响。
- 现有下载任务状态机、下载专辑交互、远程预览阅读和维护中心行为必须保持兼容。

## 目标 / 非目标

**目标：**

- 将现有下载页面重构为以漫画库为默认视图、以下载任务为并列子页签的统一工作区。
- 对当前下载目录建立可重建、可分页查询和可增量更新的本地漫画索引。
- 安全、高效地为漫画卡片和现有阅读器提供本地封面与页面。
- 提供定位、单项健康检查、回收站删除、重命名和元数据编辑，并在磁盘、索引和下载历史之间维持可恢复的一致性。
- 使大漫画库扫描、查询和阅读不会把全部资产或图片字节载入渲染进程内存。

**非目标：**

- 本次不支持多个漫画库根目录、网络盘同步、云端同步或媒体服务器协议。
- 不实现自动下载更新、在线元数据刮削或把本地元数据回写到来源站点。
- 不把 `download_history` 迁移为漫画库主表，也不改变其下载去重语义。
- 不承诺识别应用外部重命名后仍保留原资产 ID；应用内重命名必须保留。
- 不实现永久删除入口；删除只进入系统回收站。
- 不一次性解压整本漫画，也不在 Electron 中引入另一套 ZIP 解析依赖。

## 决策

### 1. 工作区保留 `downloads` 页面身份，内部拆成两个子视图

侧边栏条目改名为“漫画库”，路由/页面 ID 继续使用 `downloads`，以避免破坏页面懒创建、keep-alive、快捷状态和既有测试假设。`DownloadPage` 变成薄的工作区容器：会话级 `activeSubtab` 默认为 `library`，原任务 UI 抽到 `DownloadTasksView`，新目录 UI 放入 `LibraryCatalogView`。

下载进度监听继续位于工作区或更高层的单一 hook 中，不能因任务子视图不可见而停止。页签徽章从 `useDownloadStore` 使用 `ACTIVE_DOWNLOAD_STATUSES` 派生。漫画库筛选、布局和滚动状态保存在独立 `useLibraryStore`，侧边栏页面切换后保持。

替代方案是新增独立侧边栏“漫画库”并保留“下载管理”。这会增加一级导航、重复目录信息，也不符合用户希望升级原下载管理 tab 的意图，因此不采用。

### 2. 磁盘是真相源，新增可丢弃的 `library.db`

在应用数据目录新增 `library.db`，至少包含：

- `library_items`：`asset_id`、根目录 generation、规范化相对路径、格式、大小、mtime、标题/作者/标签/来源、页数、专辑标志、下载关联、封面键、健康摘要、metadata override、创建/扫描时间和版本。
- `library_chapters`：多章节资产的稳定章 ID、显示名、自然顺序、相对路径/压缩包前缀、页数及 manifest 版本。
- `library_reading_progress`：资产 ID、可选章 ID、当前页、总页数、最后阅读时间。
- `library_scan_state`：根目录 generation、扫描 ID、阶段、取消标记和最后成功对账时间。

页面清单只在首次阅读或资产版本变化时生成，可作为每章 JSON manifest 或规范化表持久化；其中的真实路径和压缩包条目名只在后端使用，不通过列表 IPC 暴露。索引表是缓存，任何 schema 不兼容或数据库损坏都通过隔离旧文件并完整重建恢复，禁止触碰漫画文件。

资产 ID 使用持久化随机 UUID。精确路径未变化时扫描复用 ID；应用内重命名在同一事务流程中保留 ID。外部重命名默认表现为删除旧资产并新增资产，只有当一对一未匹配项的格式、大小和轻量签名无歧义时才允许复用，禁止用全库内容哈希阻塞扫描。

替代方案一是只查询 `download_history`，但会漏掉手动导入并展示已丢失记录。替代方案二是每次打开实时全扫，随着漫画数量和压缩包大小增长会导致不可接受的延迟。二者均不采用。

### 3. 扫描器分为发现、解析、提交、对账四阶段

扩展现有 maintenance scanner 的纯函数能力，但由新的 `LibraryIndexer` 管理扫描生命周期：

1. **发现**：只枚举当前根目录顶层，拒绝隐藏项、`temp_*`、不支持项和符号链接逃逸，记录轻量 stat。
2. **解析**：只对新增或版本变化项读取压缩包目录/ComicInfo 或分析文件夹；使用有界工作池。
3. **提交**：按批次短事务 upsert 有效项，并发送扫描进度。
4. **对账**：仅在未取消且发现阶段完整成功时，删除本 generation 中未见到的陈旧索引项。

增量判定使用规范化相对路径、格式、大小和纳秒级 mtime；扫描中途取消时跳过最终删除对账，避免把未遍历项误删。完整扫描同一时间只允许一个；下载完成可提交单路径增量索引，不与完整扫描并发解析同一资产。

目录迁移完成后增加 root generation、使旧资产令牌失效并启动完整扫描；迁移失败或取消时不切换 generation。应用切回漫画库只运行轻量 discovery diff，不无条件解析所有资产。

### 4. 元数据采用明确的来源优先级与覆盖层

CBZ 字段优先级为 `ComicInfo.xml > download_history > 文件名解析 > 展示兜底`；ZIP/文件夹为 `索引人工覆盖 > download_history > 文件名解析 > 展示兜底`。人工覆盖与原始解析值分开存储，UI 能说明字段是否已写回文件。

多章节文件夹按“顶层目录是一个资产、直接子目录是章节”建模。扫描器复用自然排序，并避免专辑根和章节重复出现在漫画库。顶层单本文件夹没有章节子目录时直接生成单章 manifest。

### 5. 本地封面和页面通过按需物化缓存交付

Python 负责验证资产、读取 ZIP/CBZ 条目或文件夹图片，并把请求的封面/单页原子写入新的 `library_image_cache`。控制面 IPC 只返回内容哈希、媒体类型和资产版本；Electron 扩展 `app-image://` 的允许 kind 为 `library`，把哈希映射到后端声明的受控缓存目录，再使用 `net.fetch(file://...)` 流式交付 Chromium。

典型页面请求为：

```text
Renderer: getLibraryPage(assetId, chapterId, page, version)
  -> Python: 验证 root generation / 路径 / manifest / 页码，按需物化缓存
  <- { imageUrl: app-image://library/<sha256>, version }
Chromium -> Electron protocol -> 受控缓存文件流
```

这样图片字节不进入 JSON-RPC 或渲染进程 JS 堆，Python 仍是唯一压缩包解析者，也不会一次性解压整本。缓存键包含资产 ID、版本、章、页和条目校验信息，采用有上限的 LRU；资产版本变化时旧 URL 自然失效，后台清理旧缓存。

替代方案是 Electron 直接读取压缩包，需要新增并维护另一套归档解析和安全校验；直接向协议暴露根目录文件路径又会扩大攻击面，因此不采用。

### 6. 阅读器抽象远程与本地数据源

保留 `ComicReaderModal` 的布局、预加载和交互组件，把“获取漫画清单/获取页 URL/保存进度”抽为 reader source：

- `remote-preview` 继续调用现有 `getPreviewUrls`、`fetchPreviewImage` 和阅读历史接口。
- `local-library` 调用目录详情、页物化及本地进度接口。

本地页 URL 仍可进入现有预加载窗口，但切章、资产版本或 source 类型变化时必须清空缓存。进度按资产 ID/章 ID 节流写入，关闭阅读器和切章前立即 flush。页面请求携带版本；收到 stale-version 时停止预加载并提示重新载入。

### 7. 文件管理采用“准备令牌 + 系统操作 + 提交/对账”

所有管理入口只接收 `asset_id` 和 `expected_version`。Python 在操作时解析当前记录、重新 stat、验证 realpath 位于当前根目录，并通过资产级 mutation lock 检查下载/打包/迁移/扫描/健康检查/元数据写入占用。

删除需要 Electron 的系统回收站能力，流程为：

1. 主进程调用 Python `prepare_library_delete`，取得短期一次性 operation token 和已验证路径；路径不返回 renderer。
2. 主进程调用 `shell.trashItem(path)`。
3. 成功后调用 Python `commit_library_delete(token)`，删除索引、页 manifest/缓存，并清理路径精确匹配项；目录资产同时匹配其安全子路径下的章节历史。
4. 若步骤 3 失败，文件已经进入回收站但下次扫描会移除陈旧索引；记录 error 日志并提示“文件已移入回收站，索引将在刷新后修复”。

跨系统回收站和 SQLite 不可能形成真正原子事务，因此以“文件绝不因数据库预提交而丢失、索引可重建”为优先级。删除确认框展示后端返回的标题、类型和大小，但执行时仍重新准备，避免 TOCTOU。

定位同样由 Python 解析 asset ID 后只把路径交给 Electron 主进程，使用 `shell.showItemInFolder`/`shell.openPath`。重命名由 Python 在同一根目录内执行：先校验无冲突，移动文件，再事务更新索引和下载历史；数据库失败时尽力回滚文件名，无法回滚则立即对单项重新索引并报告部分失败。

### 8. CBZ 元数据编辑使用同目录临时文件原子替换

CBZ 编辑复用 `CBZBuilder` 的 XML 字段规则，但不能重新编码图片。后端创建同目录隐藏临时文件，逐条复制原压缩包内容、替换唯一 `ComicInfo.xml`，执行 `testzip` 和页数校验后 `os.replace`。任何失败保留原文件并清理临时文件。成功后增加资产版本、失效封面/页面缓存并刷新索引。

ZIP 和文件夹没有标准元数据容器，本次把编辑值存为索引 override，并在 UI 标记“仅应用内”。不自动新增旁车文件，以免改变用户目录结构。

### 9. IPC 契约保持单一来源和严格校验

在 `shared/types.ts` 为目录分页、资产详情、扫描状态/进度、章节与页面响应、阅读进度和管理操作定义类型，并将 Python 方法、Electron channel、preload API 和 hooks 通过现有映射保持一致。资产 ID、章 ID、页码、查询长度、筛选枚举和 operation token 都必须在 preload、主进程和 Python 权威层分级校验。

扫描与下载进度使用不同事件名，避免互相覆盖。大列表只返回必要卡片字段；详情、页 manifest 和健康结果按需请求。

## 风险 / 权衡

- **大型目录首次建库耗时较长** → 分阶段后台扫描、有界并发、分页展示、可取消；未变化资产增量复用。
- **CBZ 随机页读取仍需打开归档并解压单条目** → 页级 LRU 缓存和预加载窗口减少重复工作，禁止全本解压。
- **回收站操作与 SQLite 无法原子提交** → 文件操作成功后再提交索引；提交失败依靠可重建索引对账，绝不在回收站失败时提前删数据库。
- **mtime/size 不能强保证内容身份** → 版本令牌结合路径、stat 和 manifest 信息；检测到冲突时要求重新扫描，不用昂贵的全文件哈希阻塞目录。
- **外部重命名可能丢失资产 ID 与进度关联** → 应用内重命名保证 ID 不变；外部无歧义时尽力关联，其余作为新资产，避免误合并不同漫画。
- **编辑压缩包需要接近原文件大小的临时空间** → 编辑前检查可用空间并清晰报错；临时文件放同目录保证 `os.replace` 原子性。
- **下载完成与扫描同时触碰同一路径** → 资产级锁和 active output/temp path 集合阻止解析未完成输出，下载完成事件只在最终原子产物可见后入索引。
- **索引字段与 `download_history` 漂移** → 磁盘和索引负责当前资产，历史只做来源补全；定期扫描对账并记录失效关联，不反向伪造文件存在。

## 迁移计划

1. 增加 `library.db` schema、共享类型和只读扫描/查询 IPC，不改变现有侧边栏和下载页。
2. 接入后台首次扫描、增量索引及封面缓存，以功能开关或仅新子视图方式验证现有用户目录。
3. 将 `DownloadPage` 拆为工作区和任务视图，侧边栏改名；运行现有下载页、keep-alive 和任务状态回归测试。
4. 接入本地 reader source、页物化缓存和阅读进度，再开放“阅读/继续阅读”。
5. 最后开放定位、健康检查、回收站删除、重命名和元数据编辑；破坏性功能必须在路径与失败注入测试通过后启用。

升级不移动或重写任何现有漫画。首次访问漫画库只创建索引和缓存。回滚时可删除 `library.db` 与 library image cache 并恢复原下载页面入口，现有漫画、`download_history.db` 和下载配置保持有效；已经由用户显式执行的重命名、回收站删除或 CBZ 元数据编辑不自动逆转。

## 开放问题

- 当前方案固定单一 `downloadDir` 根目录；未来若支持多个根目录，需要把 root ID 纳入资产、缓存和操作令牌，但不在本变更中预留用户界面。
- ZIP/文件夹元数据编辑暂定为“仅应用内覆盖”。若未来需要与其他阅读器互通，应另行设计可选旁车元数据格式，而不是在本变更中私自落盘。
