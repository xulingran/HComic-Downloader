## 新增需求

### 需求:Bika 搜索页必须自动按需签到
系统必须在用户进入 Bika 搜索页且认证有效后检查当天签到状态；仅当服务端明确表示尚未签到时，才必须提交签到请求。

#### 场景:未签到用户进入 Bika 搜索页
- **当** Bika 认证有效且用户资料的 `isPunched` 为 `false`
- **那么** 系统必须通过现有 Bika Session 调用 `POST users/punch-in`
- **且** 本次签到成功后必须显示“Bika 签到成功”的成功反馈

#### 场景:用户当天已签到
- **当** Bika 认证有效且用户资料的 `isPunched` 为 `true`
- **那么** 系统禁止调用 `POST users/punch-in`
- **且** 禁止显示误导性的本次签到成功反馈

#### 场景:认证无效
- **当** 用户进入 Bika 搜索页但认证校验失败
- **那么** 系统必须显示现有登录提示
- **且** 禁止触发签到接口

### 需求:Bika 自动签到必须非阻塞且复用系统代理会话
自动签到的全部网络请求必须复用 Bika 解析器中已注入系统代理和认证信息的 Session；签到失败禁止阻塞或清空 Bika 搜索页内容。

#### 场景:签到请求失败
- **当** 用户资料请求或签到提交因网络、服务端或响应格式错误而失败
- **那么** Bika 搜索和分类入口必须保持可用
- **且** 失败必须经既有错误与日志链路保留可诊断信息

#### 场景:页面会话内重复进入 Bika 来源
- **当** 同一个搜索页挂载周期内 Bika 自动签到已经触发过
- **那么** 再次切换到 Bika 来源时禁止重复触发自动签到流程
