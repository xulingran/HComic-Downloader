## 新增需求

### 需求:可靠识别 JM 反爬挑战响应

系统必须在处理 JM HTML 响应时识别 Cloudflare 反爬挑战。识别必须优先使用 `cf-mitigated: challenge` 等明确响应元数据，并辅以稳定的挑战页标记；禁止以响应正文小于固定长度作为挑战成立的必要条件。

#### 场景:响应头明确标记挑战
- **当** JM 响应包含 `cf-mitigated: challenge`
- **那么** 系统必须将响应分类为反爬挑战，无论 HTTP 状态码和正文长度为何

#### 场景:长正文包含稳定挑战标记
- **当** JM 响应正文超过 500 字节且包含 Cloudflare 挑战平台或 “Just a moment” 等稳定标记
- **那么** 系统必须将响应分类为反爬挑战

#### 场景:正常长页面不被误判
- **当** JM 响应为正常 HTML 且不包含挑战响应头或稳定挑战标记
- **那么** 系统禁止将其分类为反爬挑战

### 需求:收藏夹挑战执行有界恢复

JM 收藏夹请求遇到反爬挑战时，系统必须复用当前 Session、系统代理、Cookie、User-Agent 和目标域名进行有界恢复；重试次数必须有固定上限，禁止无限重试或创建绕过系统代理的新会话。

#### 场景:首次挑战后重试成功
- **当** 首次收藏夹请求被识别为反爬挑战，后续有界重试返回正常收藏夹页面
- **那么** 系统必须解析并返回收藏夹内容，且不得要求用户重新登录

#### 场景:挑战持续至重试耗尽
- **当** 所有允许的收藏夹重试仍被识别为反爬挑战
- **那么** 系统必须停止请求并返回可恢复的反爬挑战错误，禁止继续无限重试

#### 场景:恢复请求保持网络上下文
- **当** 系统执行收藏夹挑战恢复
- **那么** 所有恢复请求必须使用原 JM Session 及其已注入的系统代理和认证上下文

### 需求:反爬挑战与认证失效分开报告

系统必须使用显式错误类型或等价的结构化信号区分反爬挑战和认证失效。IPC 错误映射禁止仅因错误文本包含 `403`、`cloudflare` 或 `just a moment` 就把反爬挑战报告为登录凭证失效。

#### 场景:持续挑战的用户提示
- **当** JM 收藏夹在重试后仍触发反爬挑战
- **那么** IPC 必须返回“站点人机验证/请稍后重试或检查网络与域名”语义的错误，禁止返回“登录凭证已失效”

#### 场景:明确跳转登录页
- **当** JM 收藏夹请求最终跳转到登录页或正常页面明确要求登录
- **那么** 系统必须返回需要登录状态

#### 场景:普通 403 不伪装成挑战
- **当** JM 返回 403，但既无挑战响应头也无稳定挑战页标记
- **那么** 系统必须沿用普通 HTTP 或认证错误处理，禁止将其自动分类为反爬挑战
