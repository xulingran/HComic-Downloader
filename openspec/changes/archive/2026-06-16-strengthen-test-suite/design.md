## 上下文

HComic Downloader 拥有 1435 个测试用例（Python 653 + 前端 782），覆盖率 83%。表面完备，但实际提供的发布信心远低于数字暗示。审计发现三类问题：

1. **被掩盖的真实 bug**：`test_same_drive_target_exists_reports_clear_error` 失败，根因是 `os.rename` 在非 Windows 平台静默覆盖目标文件（POSIX 语义），而代码依赖 `FileExistsError`（Windows-only 行为）。
2. **同义反复测试**：大量用例验证"mock 被调用"或"框架基本保证"，不贡献信号。
3. **危险地带测试薄**：阅读器预加载、下载状态机、IPC 契约——最近 bug 反复出现的地方——恰恰是行为测试最弱的地方。

当前架构关键约束：Electron ↔ Python 通过 JSON-RPC over stdin/stdout 通信；四个漫画来源解析器各自 `apply_system_proxy_to_session`；下载流程为 解析→下载→CBZ打包。

## 目标 / 非目标

**目标：**
- 修复 migration 跨平台文件覆盖 bug，让错误处理在所有平台行为一致
- 将测试套件从"数量驱动"转向"信号驱动"：每个用例都能回答"改了这块，敢不敢发布"
- 建立回归守护，锁定已发现的真实 bug 不再复发
- 补齐危险地带（预加载、下载、IPC）的行为验证
- 提供一条端到端冒烟，守住进程间通信边界

**非目标：**
- 不追求 90%+ 覆盖率（覆盖率衡量"执行过"而非"验证过"）
- 不给每个 store/hook 补测试（Zustand/React 基本保证无需测试）
- 不引入 Playwright 全链路 UI E2E（桌面应用 ROI 极低）
- 不重写所有 mock 测试（网络 mock 是正确的，保留）
- 不为四个解析器补全所有错误路径测试（网站改版时全挂，维护负担重）

## 决策

### 决策 1：migration bug 修复策略 —— 显式预检查而非依赖平台异常

**选择**：在 `_move_item` 调用 `os.rename`/`shutil.copytree` 前，显式 `os.path.exists(item.target)` 检查并主动抛 `FileExistsError`。

**理由**：
- `os.rename` 的异常行为是平台实现细节（CPython 文档明确标注 "If dst exists, OSError will be raised on Windows"），依赖它会耦合平台。
- 显式预检查让行为在所有平台一致、可读、可测试。
- 预检查与实际操作之间存在 TOCTOU（time-of-check to time-of-use）窗口，但迁移场景是单线程顺序执行，且用户手动触发，竞争风险可忽略。

**替代方案**：
- 改用 `os.replace`：无助于发现问题，它明确"始终覆盖"，与需求相反。
- 用 `shutil.move` 统一：它内部也依赖平台语义，问题依旧。

### 决策 2：测试精简的判断准则 —— "mock 替换测试"

**选择**：对每个断言应用唯一准则——"如果把这个 mock 换成真实实现，断言还成立吗？"

| 答案 | 判定 | 动作 |
|------|------|------|
| 是 | 行为验证 | 保留 |
| 否 | 同义反复 | 删除或重写为真实行为验证 |

**理由**：这个准则直接对应"测试是否提供了超越实现的独立信号"。它区分了"验证被测代码做了正确的事"与"验证 mock 按预期响应了自己"。

**替代方案**：
- 按覆盖率删除未覆盖代码的测试：方向相反，会误删覆盖了但无价值的测试。
- 全部保留只新增：加剧噪音，违背初衷。

### 决策 3：危险地带测试的集成深度 —— 进程内而非进程间（除冒烟外）

**选择**：
- 下载集成测试：真实文件系统（`tmp_path`），但网络层用 fixture bytes 注入而非真实 HTTP。
- IPC 契约测试：进程内实例化 `IPCServer()` 直接调用方法，验证返回结构。
- 预加载边界测试：jsdom + 真实 hook 组合，不 mock hook 内部。
- 仅冒烟测试（Phase 4）真实 spawn 子进程。

**理由**：
- 进程内调用能验证 Python 方法签名、返回结构、字段类型——这正是前后端契约漂移的高发区。
- 真实子进程开销大（启动+通信），只在最关键的冒烟里用一次，避免拖慢日常反馈。

**替代方案**：
- 全部进程间测试：最真实但测试时间从 6s 飙升到分钟级，反馈循环受损。
- 全部 mock：现状，已证明无法抓住契约漂移。

### 决策 4：IPC 契约测试的数据源 —— 从 shared/types.ts 生成预期

**选择**：IPC 契约测试的前端侧预期结构，从 `shared/types.ts` 的 TypeScript 类型推导，而非硬编码副本。

**理由**：硬编码副本会随前端类型演进漂移，让测试失去意义。从单一真相源（types.ts）推导保证测试与契约同源。

**权衡**：需要引入轻量的类型→运行时校验（如 zod 或手写 type guard）。考虑到项目目前无 zod 依赖，倾向手写最小 type guard 函数，避免新增依赖。

## 风险 / 权衡

| 风险 | 缓解 |
|------|------|
| 删除测试导致覆盖率下降，触发 CI 红线 | 本项目无覆盖率 CI 门槛；删除前记录理由；信号比数字更重要 |
| 精简误删有价值测试 | 每个删除逐条标注理由；先标记待删，确认后移除 |
| IPC 进程内测试与真实 spawn 行为有差异 | Phase 4 冒烟测试覆盖真实 spawn 路径作为补充 |
| migration 预检查的 TOCTOU 窗口 | 单线程顺序执行，用户手动触发，风险可忽略；文档标注 |
| 端到端冒烟拖慢测试（+5-9s） | 用 pytest marker 标记，可 `pytest -m smoke` 单独跑或跳过 |
| 阅读器预加载边界测试可能脆弱（依赖时序） | 测不变量（已加载页面不消失）而非具体时序 |

## 开放问题

1. **Phase 4 冒烟测试是否纳入本次变更**？它增加约 5-9s 测试时间。建议作为可选任务，由实施时根据实际复杂度决定是否启用。
2. **前端是否有最小 type guard 基础设施**？需在实施时探查是否已有运行时校验工具，避免重复造轮子。
