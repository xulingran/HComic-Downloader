## 上下文

应用的 keep-alive 页面由 `App.tsx` 中的绝对定位容器承载。虽然外层 `main` 声明了 padding，但激活页面的 `absolute inset-0` 容器仍覆盖整个可滚动区域，因此 `DownloadPage` 必须自行建立页面级留白。当前下载页根节点只有垂直间距，页头又把标题、可能很长的下载目录和三个全局操作放在同一行；任务卡虽有内部 padding，但列表整体没有居中宽度与边缘缓冲。

主窗口最小宽度为 900px，但侧边栏会占用其中一部分，因此设计仍需允许页头操作换行，并保证长目录不会挤压按钮。现有下载业务行为、任务动画和 reduced-motion 处理均已存在，本变更只调整可见结构和样式。

## 目标 / 非目标

**目标：**

- 在下载页四周建立稳定、响应式的视觉留白，并限制超宽窗口下的内容行长。
- 将页头组织为清晰的主工具栏和次级目录信息行，保证窄窗口下不拥挤。
- 统一专辑卡、普通任务卡、章节子行和空状态的间距节奏与边界层级。
- 保持所有现有按钮、筛选、任务状态、专辑折叠和动画语义不变。

**非目标：**

- 不修改 `App.tsx` 的全局页面容器，也不统一改造其他 tab 的页面边距。
- 不重新设计下载状态机、进度条、按钮文案或专辑交互。
- 不引入新的 UI 库、CSS-in-JS 方案、全局 spacing token 或运行时依赖。
- 不改变 Electron 窗口尺寸约束、IPC 契约或 Python 下载逻辑。

## 决策

### 1. 在 DownloadPage 内建立局部页面画布

下载页根层使用响应式 padding，内部内容容器使用 `w-full`、`mx-auto` 和约 1152px 的最大宽度，并保留充分的底部空间。这样可以直接解决贴边问题，且不会让其他已各自形成布局的页面发生连锁变化。

替代方案是修复 `App.tsx` 的全局 padding 语义，但这会同时改变所有 keep-alive 页面，部分页面还已有自己的侧栏或留白，容易产生双重 padding，因此不纳入本次范围。

### 2. 页头采用两层结构并允许响应式换行

第一层放置“下载管理”标题和全局操作组；第二层在存在下载目录时显示目录信息与“打开”按钮。操作组允许换行，目录文本使用 `min-w-0` 与截断显示，打开按钮保持不可压缩。主窗口较宽时仍维持紧凑横向排列，空间不足时自然换行而不横向溢出。

替代方案是保留单行并进一步缩小字号和按钮 padding，但这会降低可读性，也无法稳定处理长路径。

### 3. 使用项目现有 Tailwind 尺度形成间距层级

页面边缘在常规桌面宽度采用 24px，在较窄宽度采用 16px；页头到内容区约 24px，顶层任务卡间约 16px，卡片内部约 20px，操作控件间约 8px。具体实现复用项目已有 `p-*`、`gap-*`、`space-y-*`、颜色变量、圆角和阴影，不创建新的 CSS 变量。

### 4. 统一卡片轮廓但保留专辑强调语义

普通任务卡和专辑卡都使用项目边框颜色形成稳定边界；专辑卡继续保留左侧 accent 强调线。展开后的章节子行继续使用次级背景，但增加足够的行间距与内部留白，避免信息贴边。空状态使用同一类卡片容器和垂直居中留白，从而与有任务时的页面骨架保持一致。

### 5. 测试聚焦结构与行为回归，视觉数值由样式契约锁定

现有 `DownloadPage` 测试继续验证加载、筛选和操作行为；新增或调整测试应确认页头、目录信息、空状态和任务列表的结构存在，并验证关键操作仍可触发。样式层使用稳定的语义容器或测试标识来降低对完整 class 字符串的脆弱耦合，仅对决定布局契约的关键类做必要断言。

## 风险 / 权衡

- [最大宽度使超宽窗口不能铺满] → 下载列表以阅读和操作为主，限制行长能提高扫描效率；1152px 仍足以容纳长标题、进度条和操作按钮。
- [页头拆成两层会增加纵向占用] → 额外高度换取清晰层级与稳定响应式表现，目录行仅在配置了下载目录时出现。
- [Tailwind 类断言可能导致测试脆弱] → 优先测试语义结构与交互，只对页面 padding、最大宽度和换行等核心契约进行有限断言。
- [局部方案与其他页面间距暂不统一] → 本次严格响应下载 tab 的问题；全局页面骨架统一可作为后续独立变更评估。
