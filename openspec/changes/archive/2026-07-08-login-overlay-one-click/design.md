## 上下文

`electron/login-preload.ts` 在登录/挑战弹窗中注入一个右上角浮标叠层，承载显式的凭证提取入口。当前实现是一个四态状态机：

```
现状：  idle ──click──▶ expanded ──click「我已登录」──▶ extracting ──▶ counting
  (28px 圆点)    (220px 卡片)                          (转圈)         (倒数关窗)
                     ▲                                       │
                     └──── 失败/notLoggedIn 回退 ────────────┘
```

`idle` 态是一个 28×28 的半透明深色小圆点，带 `✓` 符号——既不显眼（小、半透明），语义又模糊（`✓` 让人以为"已经完事了"）。`expanded` 是一个纯中间态，没有承载额外信息（提示文字"登录后点此获取凭证"完全可以直接放到按钮文字里），纯粹是"为确认而确认"的二次点击。

主进程侧（`electron/login-window.ts`）的 IPC 协议不变：`LOGIN_EXTRACT`（invoke，立即返回 accepted 快响应）、`LOGIN_EXTRACT_RESULT`（主进程定向推送结果）、`LOGIN_FINISH`（倒数到 0 请求关窗）。渲染端的 `extractInProgress` / `settled` / `alreadySucceeded` 等防抖和短路逻辑全部由主进程 handler 维持，不需要改动。

## 目标 / 非目标

**目标：**
- 把触发凭证提取的交互从「两步点击」简化为「一步点击」——idle 态点击直接进入 extracting。
- 让 idle 态的触发控件视觉上更明显，用户能一眼看到入口、理解功能。
- 失败/未登录时保留可重试的展开卡片，承载后端 message，让用户知道为什么失败并能再试。
- 维持现有的所有不变量：Shadow DOM closed 隔离、固定右上角不可拖动、自带配色不依赖站点 CSS、登录/挑战双模式文案、防抖、倒数关窗。

**非目标：**
- 不改动主进程 IPC 协议和提取链（`login-window.ts` 完全不动）。
- 不改动 preload 顶层副作用、host 元素 id（`hcomic-login-overlay`）、main world prototype 补丁。
- 不引入动画系统 / framer-motion（叠层是纯 DOM + Shadow `<style>`，与 React 前端动画系统隔离）。
- 不改动计数关窗的秒数（3 秒）和兜底超时（主进程 10s）。

## 决策

### 决策 1：状态机从四态简化为四态（语义重映射）

**选择**：保留四个状态名（idle / extracting / counting + 新增 error），但**移除 `expanded` 作为触发前的中间态**。`expanded` 仅作为 `error` 态的渲染容器复用（展示错误信息 + 重试按钮）。

```
改造后：  idle ──click──▶ extracting ──success──▶ counting ──0──▶ LOGIN_FINISH
         (胶囊)             │                        │
                           fail                     cancel
                           notLoggedIn               │
                            ▼                        ▼
                          error card ◀─────────── expanded (回 expanded 态)
                        (含 message + 重试按钮)       (取消倒数回此态)
                            │ 重试 click
                            ▼
                         extracting
```

**为什么**：`expanded` 在原设计里是"idle 和 extracting 之间的确认缓冲"，但 IPC `LOGIN_EXTRACT` 的 invoke 已经返回 `accepted` 快响应、主进程侧 `extractInProgress` 防抖已足够，渲染端不需要再有一个"确认"态。胶囊按钮文字本身就表达了用户意图（"✓ 我已登录"），不需要二次确认。

**替代方案**：
- *保留 expanded 作为首次进入态，把按钮直接放到卡片里常驻*——占地大，遮挡站点内容，且小圆点→卡片的转换本来就是多余的。
- *用 confirm() 对话框做二次确认*——原生对话框丑且阻塞，违背 Shadow DOM 隔离设计。

### 决策 2：idle 态视觉从圆点改为胶囊按钮

**选择**：`idle` 态渲染一个胶囊按钮（pill button），约 `auto × 36px`，圆角 `18px`，渐变蓝色背景（`linear-gradient(135deg, #3b82f6, #2563eb)`），白色文字「✓ 我已登录」（登录模式）/「✓ 我已完成验证」（挑战模式），带柔和阴影。单击直接触发提取。

**为什么**：
- **显眼性**：胶囊带渐变背景 + 文字，视觉权重远高于 28px 半透明圆点，用户一眼就能看到入口。
- **语义自解释**：文字「✓ 我已登录 / 我已完成验证」同时承担了原 expanded 态的提示功能（"登录后点此获取凭证"）和按钮功能，省去展开步骤。
- **与错误态视觉一致**：错误时胶囊变形为卡片（同样是 `rgba(17,24,39,.92)` + `backdrop-filter`），视觉上是"按钮展开为详情"，符合用户心智模型。

**替代方案**：
- *48px 大圆点 + 脉冲动画*——占地最小，但仍需要用户理解 `✓` 符号的含义，不如文字直观。
- *常驻展开卡片*——提示文字一直可见，但 220px 卡片始终遮挡站点内容，登录/挑战页面右上角通常有站点自身的导航/按钮，常驻卡片会干扰。

### 决策 3：error 态复用 card 渲染，承载 message + 重试按钮

**选择**：失败/notLoggedIn 时，胶囊按钮位置展开为 220px 卡片，展示：
- 标题（登录助手 / 验证助手）
- ✕ 关闭（点 ✕ 回 idle 胶囊态）
- 错误 hint 文案（notLoggedIn → "未检测到登录状态"；其他失败 → 后端 message）
- 「重试」按钮（click → extracting）

**为什么**：错误反馈需要承载后端 message（可能是"验证尚未完成，请继续完成人机验证"等具体信息），胶囊按钮空间不够。复用现有 `renderCard` 渲染逻辑（已经支持 hint + btn + close）成本最低。重试按钮让用户不用关窗重来。

**替代方案**：
- *胶囊变红 + 简短错误*——占地小但信息不够，用户不知道为什么失败。
- *关窗让用户重来*——成本高，用户要重新打开窗口、重新登录。

### 决策 4：counting 态「取消」回 idle（胶囊）而非 expanded

**选择**：倒数关窗态用户点「取消」时，回到 `idle` 胶囊态（而非原设计的 expanded 卡片），因为 expanded 已经不再是触发前的常态。

**为什么**：状态机简化后，idle 胶囊态就是唯一的"待触发"态。取消倒数后回到胶囊态，用户可以再次点击触发提取，语义清晰。

### 决策 5：error 态的 ✕ 关闭按钮回 idle（胶囊）

**选择**：error 卡片的 ✕ 关闭按钮回到 idle 胶囊态（而非原 expanded 卡片下 ✕ 回 idle 圆点的对称设计）。

**为什么**：同决策 4，idle 胶囊态是唯一的待触发态。用户看到错误后可以选择重试（点按钮）或忽略（点 ✕ 回胶囊态，稍后再点胶囊重新触发）。

## 风险 / 权衡

- **[挑战模式用户可能过早点击胶囊]** → 挑战模式下用户需先在页面完成 Cloudflare 人机验证（通常 5 秒自动挑战或 checkbox）。胶囊文字「✓ 我已完成验证」隐含"先完成验证再点我"的语义；失败时错误提示「验证尚未完成，请继续完成人机验证」会引导用户。主进程 `captureJmChallengeSnapshot` 会拒绝含挑战特征的快照，叠层推回 expanded 显示具体原因，用户自然学会"先等验证完成"。这与现状的失败路径一致，只是少了一个展开步骤。
- **[idle 胶囊比圆点占地大，可能遮挡站点右上角内容]** → 胶囊约 120×36，仍远小于 220px 卡片；登录/挑战页面右上角通常是站点 logo/导航，胶囊半透明 + 可点 ✕ 收起为隐藏态（如果需要）可以缓解。但本设计暂不引入"收起为隐藏态"——胶囊本身就是入口，收起会失去意义。接受这个占地代价换取可见性。
- **[测试大改]** → `login-preload.test.ts` 现有 20+ 用例基于两步路径，全部要重写为一步路径。这是预期的重构成本，新测试要覆盖：胶囊 click 直接触发、error 卡片展示 message、重试按钮、counting 取消回胶囊态、挑战模式文案。
- **[向后兼容]** → IPC 协议、host id、preload 顶层副作用均不变，主进程无需任何改动。渲染端状态机变化对主进程透明（主进程只看到 `LOGIN_EXTRACT` invoke 和 `LOGIN_FINISH` invoke）。

## 迁移计划

纯前端 preload 改动，无数据迁移、无配置迁移、无 IPC 协议变更。部署即生效，回滚即还原 `login-preload.ts` 单文件。
