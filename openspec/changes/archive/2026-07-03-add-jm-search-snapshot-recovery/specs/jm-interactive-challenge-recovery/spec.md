## 新增需求

### 需求:搜索挑战恢复必须提供 DOM 快照兜底

当 JM 关键词搜索或首页空搜索的交互挑战恢复在验证成功后 Python 重试仍被挑战时，系统必须优先使用同一已验证浏览器窗口捕获的合格页面快照交给 Python 解析，与收藏夹快照兜底对称。快照解析成功后，系统必须将 JM 搜索恢复策略切换为后续用户主动翻页优先静默快照恢复。禁止同一次用户动作再次自动弹窗或无限重试。

#### 场景:关键词搜索 Python 重试仍被挑战
- **当** 用户完成关键词搜索验证且第一次 Python 自动重试仍返回结构化挑战，且存在来自同一可信 `/search/photos` URL 的合格页面快照
- **那么** 系统将快照连同原始搜索参数交给 Python `parse_jm_search_snapshot` 解析并返回搜索结果
- **且** 系统必须记录后续 JM 搜索可优先使用静默快照恢复

#### 场景:首页空搜索 Python 重试仍被挑战
- **当** 用户完成首页验证且第一次 Python 自动重试仍返回结构化挑战，且存在来自同一可信根路径 URL 的合格页面快照
- **那么** 系统将快照交给 Python `parse_jm_home_snapshot` 解析并返回带 sections 结构的首页结果
- **且** 系统必须记录后续 JM 首页搜索可优先使用静默快照恢复

#### 场景:搜索快照不可用
- **当** Python 重试仍被挑战且搜索快照缺失、超限、来源不可信或解析失败
- **那么** 系统停止自动恢复并返回带手动重试入口的可恢复错误
- **且** 一次用户动作不得触发第二次验证窗口

### 需求:搜索验证后静默快照翻页恢复

当 JM 搜索交互恢复已通过可信浏览器页面快照成功解析结果后，系统必须允许后续用户主动搜索或翻页优先使用隐藏浏览器快照解析，避免先触发必失败的 Python 搜索直连请求。

#### 场景:搜索快照兜底成功后进入静默模式
- **当** 用户通过可见 JM 验证窗口完成搜索恢复，且 Python 自动重试仍被挑战但浏览器页面快照解析成功
- **那么** 系统必须记录搜索静默快照恢复可用状态
- **且** 系统必须记录最近一次搜索参数（query/page/mode），供后续请求派生目标 URL 使用

#### 场景:后续搜索直接走隐藏快照
- **当** 用户主动发起 JM 搜索请求，且搜索静默快照恢复可用
- **那么** 主进程必须先使用隐藏 BrowserWindow 加载对应搜索 URL 并捕获快照
- **且** 禁止在该静默尝试前调用 Python `search`

#### 场景:搜索隐藏快照解析成功
- **当** 隐藏 BrowserWindow 捕获到来自可信 JM 搜索或首页 URL 的合格页面快照
- **那么** 系统必须调用 Python 搜索快照解析入口返回搜索结果
- **且** 禁止打开可见验证窗口

#### 场景:搜索隐藏快照失败回退
- **当** 静默搜索快照窗口超时、崩溃、加载到不可信 URL、仍处于强 challenge 页面或快照解析失败
- **那么** 系统必须回退到原有 Python `search` 和可见交互恢复流程
- **且** 禁止清除既有登录凭据和已显示搜索结果缓存

#### 场景:非交互搜索请求不触发隐藏窗口
- **当** 搜索预加载或其他非交互调用请求 JM 搜索
- **那么** 系统禁止打开可见或隐藏 BrowserWindow
- **且** 调用必须按现有非交互错误处理路径结束
