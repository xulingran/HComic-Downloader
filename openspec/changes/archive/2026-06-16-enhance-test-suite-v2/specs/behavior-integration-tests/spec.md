## 修改需求

### 需求:下载到打包必须验证真实文件系统行为

系统必须提供集成测试，用真实文件系统（而非 mock 的文件操作）验证从下载任务创建到 CBZ 文件生成的完整流转。

#### 场景:完整下载流程生成有效 CBZ

- **当** 创建一个下载任务，其图片数据由 fixture bytes 提供（而非真实 HTTP）
- **那么** 任务状态必须流转为 `completed`，且目标路径必须生成一个有效的 `.cbz` 文件，文件内必须包含正确命名的图片和 `ComicInfo.xml`

#### 场景:下载中断后恢复不损坏已有数据

- **当** 下载进行中被取消或中断，随后重试
- **那么** 已下载的分片不得损坏，最终 CBZ 必须完整且可正常解压

#### 场景:下载核心链路必须验证 URL 安全校验真实执行

- **当** 注入一个指向内网或保留地址的图片 URL，触发真实 `ImageDownloader.download` 路径（非 mock）
- **那么** 下载必须被 `UrlValidator.validate_url` 拦截，任务必须标记为失败，错误信息必须表明 URL 被安全策略拦截

#### 场景:下载核心链路必须验证代理注入真实生效

- **当** 创建下载任务并触发真实下载路径
- **那么** 用于下载的 Session 必须已应用系统代理，符合 AGENTS.md 的硬约束，验证不得依赖 mock 断言而必须检查 Session 实际代理配置

## 新增需求

### 需求:下载错误路径必须验证用户真实遭遇的故障形态

系统必须提供集成测试，验证下载链路对网络超时、损坏响应、HTTP 错误状态码等真实故障的正确处理，确保错误被捕获并以可观察方式上报。

#### 场景:网络超时被捕获并标记任务失败

- **当** 下载过程中注入网络超时（`requests.Timeout`）
- **那么** 下载必须捕获错误，任务必须标记为失败，错误信息必须可观察

#### 场景:HTTP 错误状态码被捕获并标记任务失败

- **当** 下载过程中注入 HTTP 4xx/5xx 错误响应
- **那么** 下载必须捕获错误，任务必须标记为失败
