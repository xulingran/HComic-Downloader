## 新增需求

### 需求:修复模式只重建数据库路径

修复模式必须把目标目录中匹配到的现有漫画路径写入对应的 `download_history.output_path`，禁止移动、复制、删除、重命名或覆盖任何漫画文件。修复是否成功必须只取决于目标文件在执行时仍然存在且数据库更新成功，不得要求旧 `output_path` 指向的源文件存在。

#### 场景:缺失旧路径被修复到现有目标文件

- **当** 一条历史记录的旧 `output_path` 不存在，但修复计划在目标目录中匹配到现有漫画文件
- **那么** 系统必须把该记录更新为匹配文件的路径并将条目标记为成功，且目标文件内容和位置必须保持不变

#### 场景:源路径与目标路径相同时幂等成功

- **当** 历史记录已经指向修复计划匹配到的同一个目标文件
- **那么** 系统必须将该条修复视为成功，禁止抛出“目标文件已存在”，且不得改动该文件

#### 场景:目标文件在执行前消失

- **当** 修复计划生成后、对应条目执行前，匹配到的目标文件已不存在
- **那么** 系统必须将该条目标记为失败、保留原 `output_path`，并通过既有迁移错误通知报告目标缺失

#### 场景:数据库更新失败

- **当** 目标文件存在但更新 `download_history.output_path` 失败
- **那么** 系统必须将该条目标记为失败并保留目标文件，禁止尝试通过移动文件补偿数据库错误

#### 场景:完整迁移继续执行文件移动

- **当** 迁移模式为 `full`
- **那么** 系统必须继续使用既有文件移动及目标冲突保护流程，禁止因修复模式分派而放宽覆盖保护

### 需求:待确认迁移计划必须可恢复并可放弃

迁移预览生成的 `ready` 计划必须继续视为占用态，禁止被新计划静默覆盖。前端必须能够恢复该计划；当用户从预览返回或关闭预览时，必须通过后端状态转换显式结束该计划，禁止只清除前端状态。

#### 场景:从预览返回后可重新生成计划

- **当** 用户在 `ready` 预览页点击“返回”且取消请求成功
- **那么** 后端计划必须转为 `cancelled`，前端必须回到选择页，随后生成新计划不得再收到“迁移进行中”错误

#### 场景:关闭预览后释放 ready 计划

- **当** 用户在尚未执行的预览页关闭对话框且取消请求成功
- **那么** 后端计划必须转为 `cancelled` 后再关闭对话框，禁止遗留占用中的 `ready` 计划

#### 场景:取消失败时保留可操作界面

- **当** 用户放弃预览但 `cancel_migration` 调用失败
- **那么** 前端必须保留当前预览并显示错误，禁止清空本地计划或假装取消成功

#### 场景:重新打开时恢复 ready 计划

- **当** 后端存在 `ready` 计划且用户重新打开迁移对话框
- **那么** 前端必须根据状态响应恢复迁移 ID、模式、文件数、源目录和目标目录，进入预览页并允许用户确认或取消

#### 场景:执行中关闭仍保持后台运行

- **当** 迁移状态为 `running` 且用户选择“后台运行”
- **那么** 前端必须只关闭对话框，禁止取消或替换正在执行的迁移

#### 场景:ready 计划仍禁止被直接覆盖

- **当** 后端存在尚未取消或确认的 `ready` 计划，任一入口尝试创建另一个计划
- **那么** 后端必须继续拒绝新计划，只有原计划进入终态后才能接受下一计划

### 需求:终态引擎替换必须释放资源并保持串行化

当终态迁移允许新计划创建时，系统必须在替换旧 `MigrationEngine` 前显式关闭旧引擎拥有的日志 handler。迁移生命周期使用的锁和协调字段必须只在 IPCServer 首次初始化时创建，禁止在持有旧锁期间用新锁替换它。

#### 场景:取消后新计划关闭旧引擎日志资源

- **当** `ready` 计划被取消为 `cancelled`，随后创建新计划
- **那么** 系统必须先关闭旧迁移引擎的日志 handler 再安装新引擎，旧实例不得继续占用迁移日志文件

#### 场景:新计划沿用同一迁移锁

- **当** 任一终态计划之后创建新计划
- **那么** 新计划的检查与安装必须继续使用 IPCServer 初始化时创建的同一个 `_migration_lock` 对象，禁止在临界区内替换锁

#### 场景:活动引擎不被关闭或替换

- **当** 当前迁移处于 `ready`、`running` 或 `paused`
- **那么** 新计划入口必须在替换引擎之前拒绝请求，禁止关闭活动引擎的资源
