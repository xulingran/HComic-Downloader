## 为什么

“修复数据库”模式当前会把目标目录中已经存在的漫画再次当作待迁移文件处理，导致源路径与目标路径相同时全部报“目标文件已存在”，无法完成数据库重建。同时，迁移预览生成的 `ready` 计划在用户返回或关闭对话框时不会释放，后续操作会被永久误判为“迁移进行中”，只能靠重启暂时恢复。

## 变更内容

- 将修复模式明确为纯数据库重建流程：匹配目标目录中的现有文件并更新 `download_history.output_path`，绝不移动、复制、删除或覆盖漫画文件。
- 使修复操作具备幂等性：数据库已经指向匹配文件时仍视为成功，不再产生目标冲突。
- 完善迁移预览生命周期：用户从预览返回、关闭对话框或放弃计划时，必须显式取消后端 `ready` 计划。
- 前端重新打开迁移对话框时能够识别并恢复待确认的 `ready` 计划，避免界面与后端状态脱节。
- 保持 `running` / `paused` 迁移的互斥保护，不允许新计划覆盖真实执行中的任务。
- 终态计划被新计划替换时关闭旧迁移引擎的日志资源，并保持生命周期级并发锁不变，避免 Windows 文件占用和锁串行化失效。
- 增加修复模式文件安全、数据库更新、预览取消和状态恢复的回归测试。

## 功能 (Capabilities)

### 新增功能

无。

### 修改功能

- `migration-engine`: 补充修复模式“只更新数据库、不操作文件”的行为要求，并完善 `ready` 预览计划的取消、恢复与重试生命周期。

## 影响

- Python：`migration.py`、`python/ipc/migration_mixin.py` 的修复执行路径和迁移状态管理。
- React：`src/hooks/useMigration.ts`、`src/components/settings/MigrationDialog.tsx` 的 `ready` 状态同步与预览退出处理。
- 类型契约：如有需要，调整 `shared/types.ts` 中迁移状态响应的前端映射，不新增 IPC 通道。
- 测试：`tests/test_migration.py`、`tests/test_migration_mixin.py` 及对应前端 hook / 对话框测试。
- 不改变 full migration 的文件移动、冲突保护和下载目录变更行为；不引入新依赖。
