## 上下文

迁移引擎目前用同一条 `execute()` 循环处理完整迁移和数据库修复。`plan_repair()` 生成的条目把数据库中的旧 `output_path` 放入 `source`、把目标目录中匹配到的现有文件放入 `target`，但执行阶段仍无条件调用 `_move_item()`。当数据库已经指向目标文件时，`source == target`，目标冲突保护会把它判为失败；当数据库指向缺失的旧文件时，修复也会因为源文件不存在而失败。这与“根据磁盘现状重建数据库”的语义相反。

迁移计划在预览阶段处于 `ready`。后端按既有规范将 `ready` 视为占用态，防止其他入口覆盖等待确认的计划；问题在于独立迁移对话框的“返回”和“关闭”只清空 React 本地状态，没有把计划标记为 `cancelled`。再次计划必然被后端拒绝。`syncFromBackend()` 又没有处理 `ready`，因此用户无法恢复并取消遗留计划。

本变更跨越 Python 引擎、IPC 状态响应、React hook 和对话框生命周期，但不新增 IPC 通道，也不改变完整迁移的文件安全规则。

## 目标 / 非目标

**目标：**

- 修复模式只依据目标目录中的匹配文件更新下载历史数据库，不执行任何文件移动、复制、删除或覆盖。
- 修复模式对“数据库已指向正确文件”和“数据库指向缺失旧路径”均可安全、幂等地工作。
- 目标文件在计划后消失时保留原数据库记录，并把该条目标记为失败。
- 用户放弃迁移预览时可靠释放后端 `ready` 计划。
- 对话框重新打开时恢复仍存在的 `ready` 计划，让用户可以确认或取消。
- 保持 `ready` / `running` / `paused` 的互斥保护和 full migration 的现有行为。
- 终态引擎被新计划替换时释放旧日志 handler，同时保留 IPCServer 生命周期内的同一把迁移锁。

**非目标：**

- 不改变修复模式的文件匹配算法、递归扫描范围或未匹配文件的人工解析流程。
- 不自动删除缺失的历史记录。
- 不改变完整迁移的目标冲突策略、跨盘复制策略或配置落库规则。
- 不迁移或自动修订用户当前数据库；实现完成后由用户主动再次运行修复。

## 决策

### 1. 执行阶段按迁移模式分派条目操作

保留统一的 `execute()` 状态、进度、错误收集和持久化循环，但把单条处理分为两种语义：

- `full`：继续调用现有 `_move_item()`，完成文件移动后更新数据库。
- `repair`：调用独立的数据库修复操作，确认 `item.target` 当前仍存在后直接执行 `update_output_path(item.db_key, item.target)`，不读取或操作 `item.source` 对应的文件。

这样可复用统一的取消、进度和失败统计，同时从结构上保证修复模式不会误触文件系统写操作。没有选择在 `_move_item()` 中对 `source == target` 特判，因为修复模式也必须支持 `source` 为缺失旧路径的情况；仅处理相等路径不能解决核心语义错误。

### 2. 修复成功以“目标存在且数据库更新成功”为准

`plan_repair()` 生成计划时目标文件存在，但计划与执行之间可能发生变化。执行每条修复前必须再次验证目标存在：

- 目标存在且数据库更新成功：条目标记 `done`，计入 `completed_items`。
- 目标不存在或数据库更新失败：条目标记 `failed`，保留旧数据库值并通过现有错误通知暴露。

数据库已经指向 `item.target` 时仍调用幂等更新并视为成功，避免把正确记录报告为冲突。

### 3. 保留 ready 占用规则，通过显式取消结束预览

不放宽 `_is_migration_occupied()`，也不允许新计划静默覆盖 `ready`。`ready` 可能来自设置页下载目录变更确认，也可能来自维护中心修复预览；静默覆盖会让另一处 UI 持有失效的 migration ID。

独立迁移对话框在以下行为发生时调用现有 `cancel_migration`：

- 预览页点击“返回”；
- 预览页点击关闭按钮或触发其他关闭方式；
- 预览尚未执行时组件明确放弃当前计划。

只有取消成功后才清空预览并进入选择页/关闭对话框；取消失败时保留预览和错误提示，避免再次制造前后端脱节。执行中的“后台运行”仍只关闭界面，不取消任务。

### 4. 用现有状态查询恢复 ready 预览

扩充 `MigrationStatusResponse` 的类型描述，使非 `none` 状态包含后端已返回的 `id`、`mode`、源目录和目标目录，并由状态处理器补充同盘信息。`syncFromBackend()` 返回本次取得的状态，同时维持既有 progress/complete 状态同步。

对话框打开时：

- `ready`：重建 `MigrationPlanPreview`，恢复 `mode` 并进入预览页；
- `running` / `paused`：进入执行页并展示后端进度；
- `completed`：进入完成页；
- `none` / `cancelled` / `failed`：回到选择页。

不新增专用“丢弃计划”IPC，因为现有取消操作已经提供 `ready -> cancelled` 的安全终态转换和持久化。

### 5. 以行为测试覆盖真实副作用

Python 测试使用真实临时文件和可观测的数据库替身，验证目标文件字节、源路径状态和最终 `output_path`，不只断言 mock 被调用。前端测试验证“预览返回/关闭后可再次计划”和“ready 状态恢复后可确认/取消”的用户可见行为。

### 6. 替换终态引擎时只替换资源所有者，不重建协调原语

手工冒烟发现，取消 `ready` 计划后创建新计划会调用 `_init_migration()` 覆盖旧 `MigrationEngine`，但旧实例拥有的 `FileHandler` 未关闭，Windows 会持续占用日志文件。同时 `_init_migration()` 每次都重建 `_migration_lock`；该调用发生在旧锁的 `with` 区域内，替换锁会破坏所有 plan 入口共用同一锁的串行化不变量。

`_init_migration()` 必须在替换已有终态引擎前调用旧实例的 `close()`，并只在 IPCServer 首次初始化时创建 `_migration_lock`、`_migration_thread` 和 `_migration_paused_dm`。后续接受新计划时仅替换迁移引擎，保留同一把锁和既有协调字段。没有选择让 `MigrationEngine` 依赖析构器自动释放，因为文件句柄释放时机必须确定，且 Python 垃圾回收不能提供跨实现的及时性保证。

## 风险 / 权衡

- **修复执行期间目标被外部删除** → 每条更新前重新检查目标存在，失败时不改数据库。
- **取消 IPC 短暂失败导致对话框无法立即关闭** → 保留当前预览并显示错误，让用户重试；不牺牲状态一致性换取表面关闭。
- **状态响应类型扩充影响现有 mock** → 仅对非 `none` 分支要求计划字段，并同步更新测试夹具；IPC 方法和通道保持不变。
- **统一循环仍共享 full/repair 的进度字段** → 保持既有通知契约，repair 的 phase 可继续使用现值，避免无必要的前端协议扩张。
- **已有 `ready` 状态来自设置页确认框** → 不自动覆盖；各 UI 只取消自己明确放弃的计划，并通过 migration ID 校验避免取消错误计划。
- **终态引擎替换时旧日志文件仍被占用** → 替换前显式 `close()`；测试同时验证旧 handler 释放和迁移锁对象身份保持不变。

## 迁移计划

1. 先增加失败回归测试，固定当前两个缺陷。
2. 实现 repair 条目分派与数据库专用更新路径，运行 Python 迁移测试。
3. 完善状态响应类型和 ready 同步，再实现对话框的显式取消/恢复。
4. 运行前端 hook、对话框/设置页测试以及完整提交前验证流程。
5. 若前端生命周期变更出现问题，可回滚前端部分；Python repair 分派可独立保留，不改变 full migration。
