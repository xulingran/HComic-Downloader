## 上下文

当前 `config_mixin._apply_runtime` 的 `downloadDir` applier 是 `lambda v: self._download_manager.set_output_dir(v)`——只改运行时输出目录，不碰数据库。而应用已有的「迁移」功能（`migration_mixin.handle_start_migration` → `MigrationEngine.plan_full_migration` → 后台线程 `_run_migration` → 完成回调 `_apply_runtime("downloadDir", target_dir)`）能完整移动文件 + 更新 `output_path` + 落库配置。

两条路径割裂：用户改目录时不知道要手动走迁移；走迁移时又绕过了普通的配置变更入口。本次让「改下载目录」复用迁移引擎，使两者合一。

关键约束：
- 迁移是耗时操作（大库数百 GB），不能阻塞 IPC 分发线程，必须异步。
- 迁移可能部分失败（跨盘空间不足、文件占用），失败项的处理须与现有迁移语义一致（记入 `failed_items`，不中断）。
- 配置落库的时机：必须在迁移**成功后**才写新 `download_dir`，否则迁移失败但配置已改，下载器会用新目录但文件还在旧目录。
- 前端须让用户知情并确认，不能静默移动用户文件。

## 目标 / 非目标

**目标：**
- 改下载目录时，若旧目录下存在可迁移的记录文件，自动触发 full migration（移动文件 + 更新 DB）。
- 迁移成功后才落库新 `download_dir`；迁移失败则保持旧配置，已移动文件不回退。
- 复用既有迁移进度通知与前端 UI，不新增 IPC 通道。
- 前端在改目录前提示用户"将迁移 N 个文件"，确认后才执行。

**非目标：**
- 不修改 `MigrationEngine` 本身的移动/冲突逻辑（已稳定，有独立 spec 覆盖）。
- 不实现"撤销迁移"（文件移回旧目录）——超出范围，失败项靠 `failed_items` 报告。
- 不处理旧目录与新目录无父子关系且无重叠文件的场景（如用户把目录从 D 盘改到完全无关的 E 盘新空目录）——此时旧文件不在新目录，full migration 的 `plan_full_migration` 会因 `output_path` 不在旧 source_dir 下而跳过，属合理行为。
- 不改变迁移功能的独立入口（设置页 → 迁移仍可用）。

## 决策

### 决策 1：迁移异步执行，配置落库延后到迁移完成

**选择：** 改 `downloadDir` 时，`config_mixin` 不直接落库新值，而是：
1. 调 `plan_full_migration(old_dir, new_dir)` 生成计划
2. 若计划为空（无记录或无文件需移动）→ 直接落库新值 + `set_output_dir`（保持当前快速路径）
3. 若计划非空 → 启动后台迁移线程（复用 `migration_mixin._run_migration` 的线程模型），**暂不落库**；迁移完成回调里再 `_apply_runtime("downloadDir", new_dir)` + `config.save()`

**理由：** 迁移耗时，阻塞 IPC 线程会让整个后端无响应。落库时机必须与文件实际位置一致——迁移成功前文件还在旧目录，此时若把 `download_dir` 改成新目录，下载器的新任务会写到新目录但旧资产还在旧目录，造成新的数据不一致。

**替代方案 A：** 同步执行迁移，前端显示阻塞 loading。否决：数百 GB 同步移动会让 IPC 超时，且无法展示实时进度。
**替代方案 B：** 先落库再迁移。否决：迁移失败时配置已指向新目录，文件却在旧目录，无法回退。

### 决策 2：复用 migration_mixin 的迁移编排，不重复实现线程/进度

**选择：** 把迁移编排逻辑收敛到 `migration_mixin`，暴露一个 `trigger_download_dir_migration(new_dir) -> dict` 方法供 `config_mixin` 调用，返回 `{"migrationId", "totalItems", "skipped": bool}`。内部复用 `_init_migration` + `plan_full_migration` + 后台 `_run_migration` 线程 + 完成回调（回调里完成配置落库）。

**理由：** 迁移的线程模型、进度通知、暂停/取消、状态持久化已在 `migration_mixin` 成熟实现。重复实现会分叉两套迁移语义。`config_mixin` 只负责"检测是否需要迁移 + 调用 + 把结果返回前端"。

**替代方案：** 在 `config_mixin` 内独立实现迁移线程。否决：与既有迁移功能分叉，维护成本高。

### 决策 3：前端两阶段交互——预检查 + 确认 + 执行

**选择：** 设置页改下载目录时分两步：
1. **预检查**：`setConfig` 前先调一个新的轻量 IPC（或扩展 `setConfig` 返回预检结果），后端 `plan_full_migration` 计算需要迁移的文件数 N，返回给前端。
2. **确认**：前端弹窗"检测到 N 个已下载文件在旧目录，将自动迁移到新目录并更新历史记录。是否继续？"，用户确认后才真正执行。
3. **执行**：触发后台迁移，前端复用既有迁移进度 UI（设置页或迁移页的进度条）。

**理由：** 静默移动用户文件不可接受（用户可能只是临时改目录做别的）。N=0 时跳过确认直接落库（无感）。N>0 时必须知情同意。

**简化：** 预检查与执行可合并为一次 `setConfig` 调用——后端先 `plan`，若 N=0 直接落库返回；若 N>0 则启动迁移并返回 `migrationId`，前端据 `migrationId` 是否非空决定是否展示进度。这样只需一次 IPC 往返，但牺牲了"先确认再执行"。权衡后选择**两阶段**：知情同意优先于往返效率，迁移是不可逆的大操作。

### 决策 4：迁移失败时的配置回滚

**选择：** 迁移完成回调（`_migration_complete_callback`）检查 `state.status`：
- `completed`（全部成功）→ 落库新 `download_dir`
- `partial`（部分成功，有 `failed_items`）→ **仍落库**新 `download_dir`，但前端显著提示"M 个文件迁移失败，请查看迁移日志"
- `cancelled` / 全部失败 → **不落库**，保持旧 `download_dir`

**理由：** 部分成功时，已移动的文件已在新目录，若不落库则配置与文件位置不一致。部分失败是可接受的状态（用户可后续重试失败项），不应阻断整体配置更新。全部失败/取消则文件位置未变，保持旧配置正确。

**替代方案：** 任何失败都不落库。否决：部分成功的文件已在新目录，不落库会造成新的不一致。

### 决策 5：旧目录无记录时走快速路径（无迁移）

**选择：** `plan_full_migration` 返回 `total_items == 0` 时，`config_mixin` 直接落库 + `set_output_dir`，不启动迁移线程，前端无感知。

**理由：** 首次设置目录、旧目录从未下载过东西、或用户清空了历史等场景，迁移是空操作。快速路径避免无谓的线程开销和 UI 闪烁。这也是当前行为的保真（无回归）。

## 风险 / 权衡

- **[并发] 迁移期间下载任务状态** → 迁移开始前 `toggle_global_pause` 暂停下载管理器（与现有迁移一致，`migration_mixin.py:154`），完成后恢复。防止迁移与下载同时写文件。
- **[幂等] 重复改目录** → 每次改目录都重新 `plan_full_migration`，以当前 DB + 文件状态为准。若上次迁移部分失败，本次会重新评估（已在新目录的文件不在旧 source_dir 下，自然跳过）。
- **[跨盘性能] 跨盘迁移大文件耗时长** → 复用现有迁移的进度通知与跨盘 copy 逻辑；前端进度条让用户可见耗时，可随时暂停/取消。
- **[交互复杂度] 设置页增加两阶段交互** → N=0 时完全无感（快速路径）；N>0 时一个确认对话框 + 复用既有进度 UI。复杂度可控。
- **[向后兼容] setConfig 返回结构扩展** → 新增 `migrationTriggered`/`migrationId` 可选字段，旧前端不读则忽略。Electron 主进程转发时做可选字段处理。
- **[迁移引擎 spec 边界] 不修改 migration-engine** → 本次只复用其 API，不改动其内部行为，避免触碰已有 spec 覆盖的冲突处理逻辑。
